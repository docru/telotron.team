## 1. Цель совещания

Согласовать **требования к дизайну** двух PWA — **Телотрон Pro** (тренер) и **Телотрон Client** (клиент) — до массовой отрисовки экранов и токенов в Figma. Продуктовый канон по тарифам и модулям не переопределяется; UI следует бизнес-требованиям и ТЗ §14.

---

## 2. Требования директора

### 2.1. Общие

- Дизайн на пилоте — **инструмент ежедневной работы**, не маркетинговая витрина.
- **Один бренд Телотрон**, две визуальные ветки (Pro / Client) — см. бренд-док.
- **PWA, узкий viewport (~430px)** — норма; нижняя навигация, крупные зоны нажатия.
- Юридический контур (акцепты, ПД, обратная связь) — визуально **официально**, не «мелкий футер».
- **Не в MVP дизайном:** белый лейбл Max, drag-and-drop календаря, аналитические дашборды «ради красоты».

### 2.2. Телотрон Pro

| Требование | Деталь |
|------------|--------|
| Характер | Рабочий инструмент: **структура, контроль, экспертность** |
| Центр UX | **Карточка клиента** — ФИО, пол, возраст, вес/прогресс, планы, связь с календарём |
| Календарь (M2) | Опорный модуль: month / week / day / agenda; групповые занятия различимы; отмена/перенос **с обязательной причиной** в UI |
| Инвайты (M1) | CTA «Пригласить» всегда на виду; копирование + share; партнёрская ссылка **без введения в заблуждение** |
| Офлайн | Мутации расписания только онлайн; явный баннер в workspace |
| Скорость | Минимум лишних кликов; пустой дашборд недопустим как «норма» после онбординга |

### 2.3. Телотрон Client

| Требование | Деталь |
|------------|--------|
| Характер | **Простота, поддержка, мотивация** — не «вторая CRM» |
| Якоря главной | Ближайшие занятия; **M7-lite** (быстрый дневник); скачивание планов (M5) |
| Календарь | Agenda удобен на мобиле; запись онлайн; **офлайн** — слоты тусклые, некликабельные + индикатор сети |
| Ошибки | 409 (слот занят) — понятный текст, не «немой» сбой |
| Онбординг | Инвайт → вход → короткая анкета (ФИО, пол, дата рождения), **минимум шагов** |
| Зоны | Client **визуально отличим** от Pro, но пользователь понимает: **один Телотрон** |

## 3. Требования маркетолога

### 3.1. Pro

- **Time-to-value < 10 мин:** после регистрации — профиль → первая ссылка клиенту → (желательно) слот в календаре.
- Ощущение **«всё в одном месте»** (карточка + календарь + планы — одна система, не три приложения).
- **«Пригласить клиента»** — самый заметный CTA на списке клиентов (ТЗ §14.3).
- Тарифы читаются по **функциям**, не по «серым кнопкам»; апгрейд с понятной выгодой.
- Партнёрская ссылка — **вторичный** блок, не конкурирует с клиентским инвайтом на первом экране.
- **Скорость PWA** важнее оттенка синего.

### 3.2. Client

- Клиент **не платит** — интерфейс лёгкий, ощущение **подарка от тренера**, не рекламы платформы.
- **Один экран в день** для M7-lite — без глубоких вложенных меню.
- Прогресс виден **данными** (вес, отметки), не декоративными картинками.
- Офлайн формулируется как «запись при сети», не как «сломалось приложение».
- После первого успешного входа — понятный призыв **установить PWA** на экран.

### 3.3. Визуальное позиционирование

- **Pro** — экспертность и контроль (синяя ветка).
- **Client** — лёгкость и прогресс (зелёная ветка).
- **Не смешивать** ветки: клиент не должен чувствовать, что его заставляют «работать как тренер».

## 4. Предложения дизайнера (приняты к работе)

### 4.1. Дизайн-система

- Общая типографика и радиусы; два набора токенов: **`theme-pro`** / **`theme-client`** (CSS-переменные поверх shadcn). HEX и правила — [Токены цветов — Pro и Client](Токены%20цветов%20—%20Pro%20и%20Client.md).
- Семантика: `foreground`, `muted`, `destructive`, `warning`, `offline`; для календаря — отдельные токены слотов/типов занятий.
- Pro: body 14–15px, плотные списки; Client: body 16px, кнопки min **44px** высота.
- Иконки: единый набор (Lucide), разный акцент цвета по зоне.

### 4.2. Pro — приоритетные экраны

- **Shell:** нижний tab bar — Клиенты · Календарь · Ещё (инвайты, планы, настройки).
- **Список клиентов:** sticky CTA «Пригласить»; карточка — инициалы, имя, «следующее занятие».
- **Карточка клиента:** шапка; кликабельный вес → модалка прогресса; блоки Планы / Занятия.
- **Календарь:** month по умолчанию; индивид / группа — разная кодировка; клик → sheet (состав, отмена с причиной).
- **Инвайты:** две карточки-ссылки (клиент / партнёр); primary — «Скопировать» + share.

**Референс hi-fi:** список клиентов + календарь (неделя).

### 4.3. Client — приоритетные экраны

- **Главная:** приветствие; «Сегодня» (M7-lite); «Занятия»; «Мои планы».
- **Календарь:** agenda-first; свободные слоты — outline; offline — opacity ~40% + иконка «нет сети».
- **Запись:** tap → bottom sheet; 409 → toast + подсказка.
- **M4:** карточка «Подтвердить» на главной, одна кнопка.
- **Онбординг:** до 3 шагов без длинного wizard.

**Референс hi-fi:** главная (online + offline на одном макете).

### 4.4. Ближайшие deliverables (2 недели)

1. Figma: токены + базовые компоненты (кнопка, карточка, sheet, toast, offline-banner).
2. Hi-fi: 4 экрана выше + empty / error / loading skeleton.
3. Компонент **«функция недоступна на тарифе»** (без отдельного скина Light).
4. Не полный каталог §14 — только **приоритетные 6 экранов**, остальное наследует shell.

## 5. Ответы на открытые вопросы дизайнера

### 5.1. Light / Профи / Максимальный — что рисуем в MVP?

**Решение (директор + маркетолог):**

| Тариф | UI на пилоте |
|-------|----------------|
| **Лайт** | Тот же shell **Телотрон Pro**. Отличие — **нет групп / групповых занятий**; gate «доступно на Профи», **без** водяных знаков на каждом экране. |
| **Профи** | Полная операционка, тот же shell. |
| **Максимальный** (+ 2 мес. промо) | Тот же shell + слой «рост» по мере реализации в ТЗ. **Отдельная «премиум-тема» не рисуется.** |
| **Брендирование Max** | Зарезервировано только за Максимальным, **вне MVP** ([матрица-тарифов-платформа.md](../../../_telotron.ru/docs/Бизнес-требования/00-канон-mvp/матрица-тарифов-платформа.md)). В Figma — слот post-MVP, без реализации. |

- **Не** три скина Light / Pro / Max.
- **Не** водяные знаки «Light» на календаре и карточках.
- Промо 2 месяца — визуально **полноценный продукт**; урезание Light — через **понятные gate’ы**, не «наказующий» серый UI.
- **Client** не отражает тариф тренера.

### 5.2. Иллюстрации vs фото (Client)

**Решение:**

- **Приоритет — данные** (списки, цифры, статусы, мини-график веса).
- **Без** стоковых фото людей и без «до/после» на пилоте.
- **Абстракт** — только: empty state (3–4 варианта), офлайн, один блок онбординга «вы в программе тренера X» (без фото тренера в MVP).
- Аватары — **инициалы**, не обязательное фото пользователя.
- Мотивация — **микрокопи + прогресс** («−1,2 кг за месяц»), не декоративный «бегун».

### 5.3. «Ты» или «вы»?

**Решение:**

| Контур | Обращение |
|--------|-----------|
| **Pro** | **«Вы»** — деловой тон, ошибки и подтверждения |
| **Client** | **«Ты»** — сопровождение, близость к тренеру |
| **Публичная витрина, юридические тексты** | **«Вы»** (согласование с юристом) |
| **Admin** | **«Вы»** |

**Уточнения:**

- Pro: в **одном** welcome-блоке допустим мягкий тон, далее везде «вы».
- CTA без местоимения: «Пригласить клиента», «Записаться», «Подтвердить».
- В Figma — два мини-гайда **Voice Pro** и **Voice Client** (5–10 типовых фраз: ошибка, empty, успех, офлайн, тариф).

## 6. Итоговая таблица решений

| Вопрос | Решение |
|--------|---------|
| Скины по тарифам | **Один** UI Pro + **один** Client; тариф = функции + copy |
| Max-брендирование | **После MVP** |
| Иллюстрации Client | Данные в приоритете; абстракт только empty / онбординг / офлайн |
| Тон | **Pro — вы**, **Client — ты** |
