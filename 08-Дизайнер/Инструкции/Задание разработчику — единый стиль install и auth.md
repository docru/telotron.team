# Задание разработчику: единый стиль install + auth (Pro / Client)

**От:** дизайнер (08)  
**Статус:** к передаче в разработку  
**Связь:** [Система стилей — Pro и Client (MVP)](Система%20стилей%20—%20Pro%20и%20Client%20(MVP).md) · [Проверка текстов](Проверка%20текстов%20—%20список%20исправлений%20для%20разработчика.md)

---

## 1. Проблема (согласовано с заказчиком)

Сейчас **два разных визуальных мира**:

| Контур | Файлы | Как выглядит |
|--------|-------|--------------|
| **Публичные blade-страницы** | `views/pro/layout.blade.php`, `views/client/layout.blade.php`, `install.blade.php`, `login.blade.php` | Широкая «маркетинговая» карточка (~640px), градиентный backdrop, noise, **свой** inline-CSS, кнопки `.btn` / `.btn-primary` |
| **SPA auth** | `LoginPage.vue` (вход + восстановление), `RegisterPage.vue` в `TelotronAppShell` (`meta.public`) | Узкая колонка **430px**, фон `bg-background`, типографика Geist / `telotron-*`, кнопки pill из дизайн-системы, **без логотипа** (только иконка Lucide в заголовке) |

Пользователь переходит **install → login (blade) → /workspace/login (SPA)** и видит **три разных оформления**. Это ломает ощущение «один Телотрон» и узнаваемость бренда.

**Логотип:** на install/login (blade) есть `/brand/logo.png`; в SPA вход/регистрация/восстановление — **нет**, только текст «Телотрон» в shell (и то header на public-роутах **скрыт**).

---

## 2. Цель

1. **Один визуальный каркас** для всех точек входа: install, login (включая восстановление), register (Pro и Client).
2. **Логотип Телотрон** (единый знак из бренда) на каждом из этих экранов + подпись **Pro** / **Client**.
3. Стили из **канона MVP**: `theme-pro` / `theme-client`, Geist, `telotron-page`, Card, кнопки `variant="pill"`, max-width **430px** — как в [Система стилей §8](Система%20стилей%20—%20Pro%20и%20Client%20(MVP).md).
4. **Не** дублировать тяжёлый marketing-backdrop на auth (градиент/noise/glass 640px) — публичный контур должен совпасть с приложением, а не с лендингом welcome.

---

## 3. Решение (рекомендуемое направление)

### 3.1. Общий layout «публичная зона»

Вынести один компонент (предпочтительно Vue, чтобы install тоже можно было свести к одному стилю):

**Вариант A (предпочтительно):** `TelotronPublicLayout.vue`

- Используется в `LoginPage`, `RegisterPage` (восстановление — режим на `LoginPage`, не отдельный URL).
- Для blade: минимальный `workspace` host + тот же layout через Vue, **или** blade только редиректит на `/workspace/login` (install остаётся blade с **теми же** CSS-классами из `app.css` — см. вариант B).

**Вариант B:** blade `pro.layout` / `client.layout` для install+login **упростить**:

- Убрать backdrop/noise/широкую 640px card.
- Подключить только `app.css` + классы как в SPA: `theme-pro`/`theme-client`, обёртка `max-w-[430px]`, компонент шапки с логотипом.
- Ссылки «Войти» ведут на `/workspace/*` без смены «мира».

**Выбор варианта — на усмотрение разработчика**, критерий приёмки — §5.

### 3.2. Шапка с логотипом (обязательно)

На **каждом** экране install, login, register, recovery:

```
┌─────────────────────────────┐
│  [logo 40px]  Телотрон      │
│               Pro | Client  │  caption, muted
└─────────────────────────────┘
```

- Изображение: `/brand/logo.png` (или SVG, когда появится).
- `alt="Телотрон"`.
- Подпись зоны: `Pro` / `Client` — как в `TelotronAppShell`, но **с картинкой**, не только текст.
- На public SPA **показывать эту шапку** даже когда `showShell === false` (отдельный `TelotronPublicHeader`, не полный shell с tab bar).

### 3.3. Что убрать / не смешивать

| Убрать с auth/install | Оставить |
|------------------------|----------|
| Inline `--pro-bg`, отдельные `.btn` в layout blade (дубль shadcn) | `--primary`, `--background` из `theme-*` |
| Backdrop + noise на auth | Плоский `bg-background` (#F8FAFC Pro / #FFF Client) |
| Ширина card 640px на auth | `max-w-[430px] mx-auto` |
| Иконка Lucide вместо логотипа в h1 (Login, Register, Recovery) | Логотип в шапке; h1 — только «Вход» / «Регистрация» / «Восстановление доступа» |
| Дублирующая карточка «Добавить…» **вверху** auth без иерархии | Баннер install по §3.4 (primary «Установить») |

### 3.4. Приоритет установки PWA (продукт + маркетолог)

**Цель:** если пользователь открыл сайт **в браузере** (не standalone), вести к установке ярлыка. Сейчас наоборот: на `/install` primary — «Продолжить без установки», на auth — едва заметная «Добавить на главный экран».

#### Слово «Установить» vs «Добавить на главный экран»

| | **Установить** | **Добавить на главный экран** |
|--|----------------|-------------------------------|
| Длина | Коротко, глагол | Длинно, жаргон браузера |
| Client (средний человек) | Понятно | Часто непонятно |
| Android Chrome | Совпадает с системным UI | Реже в меню |
| iOS Safari | Не дословно (там «На экран Домой») | Точнее технически |

**Решение дизайна:**

- **Кнопки и ссылки:** везде **«Установить»** (primary).
- Подпись под кнопкой или на странице `/install`: одна строка пользы — *«Откроется с главного экрана, как обычное приложение»*.
- На `/install` в списке инструкций **оставить** точные шаги для iOS: «Поделиться → На экран „Домой“» (это не заголовок кнопки, а помощь).

Не использовать в primary: «Добавить на главный экран», «Продолжить без установки» (звучит как «установка — лишнее»).

#### Иерархия CTA

**Страница `/install`**

| Уровень | Элемент | Текст |
|---------|---------|--------|
| **Primary** | `Button` pill, full-width | **Установить** |
| Действие | `beforeinstallprompt` / скролл к инструкции, если prompt недоступен (iOS) | — |
| **Secondary** | `Button` outline или text link | **Продолжить в браузере** → login |
| Не делать | Primary «Продолжить без установки» | — |

**Login / Register / Recovery (в браузере, не standalone)**

Пока `display-mode: browser` и пользователь ещё не dismiss’ил баннер (localStorage, см. текущий skip на install):

| Уровень | Элемент | Текст |
|---------|---------|--------|
| **Баннер** под шапкой с логотипом | `TelotronCard` subtle или accent border | Заголовок: **«Установите приложение»** (Pro: «вы» / Client: «Установи приложение»). Подпись: зачем (занятия, клиенты — одна строка по зоне). |
| **Primary в баннере** | ссылка на `/install` или inline install flow | **Установить** |
| **Secondary** | текст под формой входа | **Продолжить в браузере** — не отдельная огромная кнопка; основная форма остаётся главной после баннера |
| После dismiss баннера | ссылка внизу auth (`TelotronInstallFooterLink`) | **Установить на телефон** (не короткое «Установить» — яснее вне баннера) |

После **первого успешного входа** (маркетолог): повторить баннер на главной workspace один раз — отдельная задача, но тот же copy «Установить».

**Blade `/login`:** те же приоритеты; не три равные ghost-кнопки — **Установить** (primary) → Войти (secondary) → Регистрация / Восстановление (links).

#### Логика показа

- Если уже `standalone` / `fullscreen` / `navigator.standalone` — **не показывать** баннер install на auth (как сейчас редирект с install на login).
- Ключ dismiss: например `telotron_{zone}_install_banner_dismissed` — по продуктовому решению.

### 3.5. Тексты

Правки микрокопи — по [Проверка текстов — список исправлений](Проверка%20текстов%20—%20список%20исправлений%20для%20разработчика.md) (раздел Auth). В этом задании — **визуал, структура и отображение OTP**.

### 3.6. Шаг OTP (вход и восстановление) — `LoginPage`

Референс: ревью экрана восстановления (май 2026). Те же правила для **login** после Passkey, если показывается код.

#### 3.6.1. Срок действия кода (P0)

**Сейчас (нельзя):** `Код отправлен через Telegram (до 2026-05-21T16:53:40+00:00)`.

**Нужно:** человекочитаемый срок, без ISO из API.

| Вариант | Пример |
|---------|--------|
| Предпочтительно | «Код отправлен в Telegram. Действует **10 минут**» (если TTL фиксирован) |
| Альтернатива | «Код отправлен на e-mail. Действует до **19:53**» — `expires_at` в **локальной TZ** пользователя, формат `Intl.DateTimeFormat('ru-RU', { hour: '2-digit', minute: '2-digit' })` |

Вынести форматирование в общий helper, например `formatOtpDeliveryMessage(channel, expiresAt)` — использовать в Login и Recovery.

#### 3.6.2. Debug OTP (P0 на prod/stage для пилота)

**Сейчас:** жёлтый блок «Заглушка доставки (код на экране): 153338».

**Нужно:**

- Показывать **только** если в ответе API есть `debug_otp` **и** среда разработки (например `import.meta.env.DEV` или флаг с бэкенда `APP_DEBUG` — согласовать с бэкендом).
- На пилоте и prod пользователь **не видит** код на экране.

#### 3.6.3. Объём текста над полем (P1)

**Сейчас:** два абзаца `recoveryLead` + отдельный абзац про смену номера + строка канала доставки + (заглушка).

**Нужно (Pro):**

```
Восстановление доступа

Новый телефон или не работает Face ID? Укажите номер, с которым 
регистрировались — пришлём код.

[статус: Код отправлен в Telegram. Действует 10 минут]

Код из сообщения
[      000000      ]
```

- Про **смену номера** — одна строка **внизу** экрана, muted + ссылка: «Другой номер? Смена — в поддержке после входа» (не второй абзац под заголовком).
- `RecoveryPage`: убрать дублирование, если `recoveryLead` в `voiceCopy` уже содержит смысл; caption про поддержку — отдельное поле `recoveryPhoneHint`.

**Client:** тот же layout, «ты» в текстах (`voiceCopy.client`).

#### 3.6.4. Фокус в поле кода (P1, UX)

При переходе на шаг OTP (`step === 'otp'` в Login/Recovery, `registrationPhase === 'otp'` в Register) **сразу** ставить фокус в input кода (`ref` + `nextTick` → `focus()`), чтобы на мобильном открылась цифровая клавиатура без лишнего тапа.

#### 3.6.5. Мелочи (P2)

- Убрать лишний `<hr>` перед «Вернуться ко входу», если есть «Изменить номер».
- Кнопки OTP: «Подтвердить код» / «Отправить снова» — ок; на login после унификации — те же подписи или «Войти» / «Отправить код снова».

#### 3.6.6. Файлы

- `LoginPage.vue` (режимы входа и восстановления, шаг OTP)
- `RegisterPage.vue` — тот же формат срока для шага кода, если есть `otpExpiresAt`
- `shared/format-otp-ui.ts`, `shared/use-otp-code-focus.ts` (или аналог)

## 5. Критерии приёмки

- [ ] Pro: install, login, register, recovery визуально **одного семейства** с экраном «Клиенты» после входа (шрифт, цвет primary, радиусы кнопок, ширина).
- [ ] Client: то же с зелёным primary.
- [ ] На всех четырёх типах экранов (install + 3 auth) виден **логотип** `/brand/logo.png` и подпись **Pro** / **Client**.
- [ ] Нет широкой 640px «стеклянной» карточки на SPA auth.
- [ ] Нет перехода «градиентный фон → плоский серый → другой плоский» без логотипа.
- [ ] `theme-color` в manifest/layout не ломается (Pro `#1D4ED8`, Client `#16A34A`).
- [ ] Mobile ~430px: шапка и форма не обрезаются, touch targets ≥40px Pro / ≥44px Client.
- [ ] В браузере: **primary «Установить»**, не «Продолжить без установки»; на auth баннер заметнее формы «добавить на экран» мелким текстом.
- [ ] В standalone PWA баннер установки на auth не показывается.

### 5.1. Шаг OTP (Login / Register / Recovery) — по §3.6

- [ ] **P0:** нет ISO-даты `2026-05-21T16:53:40+00:00` в интерфейсе; срок кода человекочитаемый.
- [ ] **P0:** блок `debug_otp` / «Заглушка доставки» не виден на prod и пилоте.
- [ ] **P1:** Recovery — не более 2 коротких абзацев над полем кода; смена номера — ссылка внизу, не второй абзац в шапке.
- [ ] Login / Register: тот же формат строки «Код отправлен…».
- [ ] **P1:** при показе шага OTP фокус сразу в поле «Код из сообщения» (клавиатура на телефоне).
