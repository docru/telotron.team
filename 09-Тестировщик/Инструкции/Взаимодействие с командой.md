# Взаимодействие с командой

## Роли в цикле качества

| Роль | Вклад |
|------|--------|
| **Тестировщик (09)** | Чеклисты, приёмка, регрессия, go/no-go рекомендация, сценарии E2E |
| **Разработчик (03)** | Реализация, **feature/Vitest-тесты**, фиксы, bump `build.*` при PWA |
| **Директор (01)** | Scope пилота, приоритет дефектов, финальный go/no-go |
| **Сисадмин (04)** | Стенд, домены, CI, доступы |
| **Поддержка (06)** | Обращения пользователей, первичная информация |
| **Юрист (07)** | Тексты оферт и акцептов (не тестирует UX вместо QA) |
| **Дизайнер (08)** | Макеты и визуал; QA проверяет **соответствие реализации** и читаемость |

## Эскалации

| Ситуация | Кому |
|----------|------|
| Блокер перед пилотом | Директор + разработчик |
| Нужен стенд / данные | Сисадмин + разработчик |
| Спор «баг или ТЗ» | Директор |
| Юридический текст на экране | Юрист (через директора) |
| Этика продукта | Директор → при необходимости священник (00) |

## Ритм (рекомендуемый)

- **Старт спринта / модуля:** тестировщик получает scope и ссылки на ТЗ.
- **Готовность к приёмке:** разработчик сообщает «можно принимать» + зелёные автотесты по модулю.
- **Приёмка:** по [Приёмка модуля](Приёмка%20модуля.md).
- **Перед релизом:** [Регрессия перед релизом](Регрессия%20перед%20релизом.md).

## Разделение с разработчиком по автотестам

| Автотесты | Ответственность |
|-----------|-----------------|
| `tests/Feature/Api/*`, доменная логика | Разработчик |
| `npm run test:ts` (утилиты, парсеры) | Разработчик |
| E2E 1–3 потока (когда появится) | Совместно; сценарии — тестировщик |
| Ручная регрессия MVP | Тестировщик |

Тестировщик **блокирует релиз** при красных `php artisan test` / согласованном наборе Vitest, даже если не писал тесты сам.
