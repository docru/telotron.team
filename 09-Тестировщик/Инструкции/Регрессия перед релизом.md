# Регрессия перед релизом

## Когда делать

- Перед **демо** для директора/команды.
- Перед выкладкой на **пилотный** стенд или внешних тестировщиков.
- После **крупного merge** (workspace, auth, календарь, инвайты).
- Не реже **раза в спринт** по критичному набору, если идёт активная разработка.

## Объём

1. Полный прогон [Критичные сценарии MVP](Критичные%20сценарии%20MVP.md) (отмечать дату и стенд в шапке файла или в комментарии к прогону).
2. **Smoke PWA:** `GET /version.json` на Pro и Client, вход в workspace, один сценарий Pro + один Client.
3. Прогон **автотестов** в контейнере: `php artisan test`, `npm run test:ts` — без красных тестов на релиз (или явный tech debt с директором).

## Время (ориентир)

- При живом чеклисте и готовом стенде: **2–4 часа** на полный критичный набор.
- Расширенная регрессия (новые модули питания, трекер, календарь v2) — отдельная сессия по плану в `_telotron.ru/docs/Техдок/00-мета/план-реализации-mvp.md` §12.

## Go / No-go

**Go** — критичные сценарии пройдены, блокеров нет, автотесты зелёные.

**No-go** — есть блокер по безопасности, потере данных, входу, оплате пилота, сквозной цепочке тренер–клиент.

**Go с оговорками** — пилот допустим, перечень известных дефектов и обходов согласован с директором (не для скрытия блокеров).

Решение фиксирует **директор**; тестировщик даёт **рекомендацию** и факты.

## После релиза

- Обращения из **поддержки**: классификация (регрессия / известный / не баг / ТЗ).
- При регрессии — срочный тикет разработчику + при необходимости пункт в чеклист до фикса.
