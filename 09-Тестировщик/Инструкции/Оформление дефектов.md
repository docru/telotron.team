# Оформление дефектов

## Когда заводить

- Фактическое поведение **не совпадает** с согласованным ТЗ / техдоком / принятым чеклистом.
- **Регрессия:** работало, перестало после изменения.
- Риск **безопасности** или **потери данных** (всегда блокер до согласования с директором).

Не заводить как дефект без согласования: **новая фича**, не входящая в scope приёмки; **изменение ТЗ** — эскалация директору.

## Где хранить

- Файлы: `Команда/09-Тестировщик/Дефекты/открытые/` → после verified в `закрытые/`.
- Именование и статусы: [Дефекты/README.md](../Дефекты/README.md).
- Копировать тело из [_шаблон.md](../Дефекты/_шаблон.md).
- GitHub Issues (опционально): одно описание в issue, в файле — ссылка `Issue #N`.

## Приоритеты (для директора)

| Уровень | Примеры |
|---------|---------|
| **Блокер** | Нельзя войти, потеря данных offline/outbox, чужие данные видны, пилот невозможен |
| **Высокий** | Основной сценарий модуля сломан, 409/422 без сообщения пользователю |
| **Средний** | Обход есть, косметика важная для пилота |
| **Низкий** | Мелочь UI, редкий край |

## Жизненный цикл

1. Тестировщик заводит → **разработчик** исправляет + по возможности **feature-тест**.
2. Тестировщик **перепроверяет** на том же стенде.
3. При закрытии — пункт в регрессии остаётся в [Критичные сценарии MVP](Критичные%20сценарии%20MVP.md).

Обращения от **поддержки** дублировать в тот же формат, если это баг продукта.
