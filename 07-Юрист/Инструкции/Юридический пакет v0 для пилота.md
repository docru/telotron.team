# Юридический пакет v0 для пилота

**Назначение:** единый перечень документов и тем, которые должны быть **подготовлены как черновики (v0)**, согласованы с юристом и опубликованы до выхода на **ограниченный пилот** (реальные тренеры и клиенты, без массовых продаж и без встроенного биллинга, если иное явно не запланировано).

**Не является юридической консультацией.** Итоговые формулировки и выбор правовых конструкций — зона ответственности профильного юриста (IT, 152-ФЗ, договорное право).

## 0. Предусловие: уведомление Роскомнадзора (gate)

**Пакет v0 и пилот на реальных ПДн — только после подачи уведомления об обработке персональных данных (оператор) в Роскомнадзор** (или иного законного основания, если юрист подтвердит исключение).

Подключение тренеров и клиентов (период I, outreach с **01.06**) **запрещено** до закрытия gate. Критерии и журнал: [уведомление-роскомнадзор-оператор-пд](../../../_telotron.ru/docs/Бизнес-требования/04-комплаенс-и-аудит/уведомление-роскомнадзор-оператор-пд.md).

**Связанные материалы:**
- карта рисков и минимальные меры: `юридические риски.md`;
- календарь созревания документов (в т.ч. переход v0 → v1.0): `Документы.md`;
- продуктовый объём пилота: `План реализации — этапы.md` (этап 1), `MVP-1.md`.

## 1. Что считается «пакетом v0»

| Критерий | Смысл |
|----------|--------|
| **v0** | Полный каркас тем, структура разделов, заполненные **фактами о продукте** (кто оператор, какие данные собираются, где хостинг, контакты DPO/ответственного). Тексты готовы к **ревью юриста**; допускаются пометки «уточнить у юриста». |
| **После ревью** | Фиксируется редакция **v1.0** (или иная нумерация по договорённости), дата вступления в силу, публикация на сайте/в приложении и **журнал версий** (см. `Документы.md`). |
| **Граница пилота** | Пилот без приёма оплат через платформу: **не** подменяет полноценный платёжно-чековый пакет; достаточно ясных **правил сервиса** (роль платформы, границы ответственности, запреты по контенту). При появлении оплат — отдельный подэтап с юристом (оферта, возвраты, чеки), как в плане реализации. |

## 2. Состав пакета v0 (обязательные документы)

Ниже — **отдельные правовые тексты** (или логически обособленные разделы одного «соглашения», если юрист упростит структуру). Для каждого указано: зачем, что обязательно раскрыть с точки зрения продукта, где показывать пользователю.

### 2.1. Политика конфиденциальности

| Поле | Содержание |
|------|------------|
| **Зачем** | 152-ФЗ: прозрачность обработки ПДн, права субъекта, контакты оператора. |
| **Оператор** | Юр. лицо / ИП, ИНН, адрес, контакт (e-mail), ответственный за обработку. |
| **Категории данных** | Идентификация (ФИО, телефон, e-mail), учётные данные, **данные о здоровье/самочувствии, замеры, фото**, переписка в чате, технические логи (IP, user-agent — по факту сбора). |
| **Цели** | Оказание функций сервиса (CRM, планы, отчётность, напоминания, чат), безопасность, поддержка, выполнение требований закона. |
| **Правовые основания** | Заполняет юрист (согласие, договор, иные основания по факту модели). |
| **Трансграничность / хостинг** | Где физически хранятся данные (РФ / иное — **фактически**). |
| **Сроки хранения** | В т.ч. чат, логи, удалённые аккаунты — принципы (детали — с юристом). |
| **Права субъекта** | Доступ, уточнение, удаление, отзыв согласия, жалоба в РКН — в формулировках юриста. |
| **Передача третьим лицам** | Хостинг, аналитика (если есть), почта, Sentry и т.д. — перечислить **реально используемые** сервисы. |
| **UI** | Ссылка в подвале, при регистрации, в настройках; версия и дата редакции на странице документа. |

### 2.2. Пользовательское соглашение (правила сервиса)

| Поле | Содержание |
|------|------------|
| **Зачем** | Правила пользования платформой, акцепт, ограничение ответственности оператора в рамках модели «сервис для тренера», а не исполнитель услуг клиенту. |
| **Обязательно** | Описание сервиса; регистрация и аккаунт; **роли** (тренер, клиент, менеджер — если есть в пилоте); запреты (незаконный контент, оскорбления, спам); интеллектуальные права на контент пользователей; блокировка/удаление аккаунта; применимое право и споры (блок для юриста). |
| **Связь с тренером и клиентом** | Явно: договор на оказание услуг **между тренером и клиентом** не заменяется платформой; платформа — инструмент (формулировка для юриста). |
| **Чат и контент** | Минимум: правила поведения, запрещённые темы, что оператор может удалить/ограничить; сроки хранения — см. также п. 2.5 при необходимости отдельного документа. |
| **UI** | Акцепт при регистрации + постоянная ссылка; версия документа. |

### 2.3. Согласие на обработку персональных данных

| Поле | Содержание |
|------|------------|
| **Зачем** | Явное согласие на обработку ПДн, перечень действий и целей в объёме, требуемом законом. |
| **Продукт** | Чекбокс (нельзя предустановить «вкл») + ссылка на политику; фиксация факта и времени акцепта в системе (аудит). |
| **Содержание** | Перечень обрабатываемых данных и целей — согласован с политикой; финальный текст — юрист. |

### 2.4. Отдельное согласие на обработку специальных категорий / чувствительных данных

| Поле | Содержание |
|------|------------|
| **Зачем** | В `юридические риски.md`: замеры, самочувствие, фото, данные о здоровье — повышенный риск; отдельное согласие как минимальная мера. |
| **Продукт** | Отдельный чекбокс при первом входе в функции, где собираются такие данные, или при регистрации клиента — **как решит юрист**; нельзя «прятать» внутрь общего текста без явного акцепта, если юрист требует раздельности. |
| **UI** | Явная формулировка, что именно разрешается; ссылка на политику. |

### 2.5. Дисклеймер «сервис не является медицинским»

| Поле | Содержание |
|------|------------|
| **Зачем** | Снижение риска квалификации деятельности как медицинской (`юридические риски.md`, §2, §8). |
| **Содержание** | Сервис не ставит диагнозов, не назначает лечение, не заменяет врача; рекомендации носят информационный/образ жизни характер; при противопоказаниях — консультация врача (детали — юрист). |
| **UI** | Отдельный экран/блок при первом входе клиента + краткое напоминание в подвале или в настройках; для тренера — при публикации планов (гайд + ссылка на полный текст). |

### 2.6. Правила использования чата (для пилота с чатом из MVP-1)

| Поле | Содержание |
|------|------------|
| **Зачем** | `юридические риски.md` §6; в `Документы.md` к **25.05** указаны финальные правила чата — для v0 достаточно **сжатого черновика**, который юрист включит в ПС/соглашение или выделит отдельно. |
| **Темы** | Запрещённый контент; доступ только участникам связки; модерация/удаление по жалобе; хранение и удаление переписки (ориентиры). |
| **UI** | Ссылка из чата («правила») + акцепт при первом отправлении сообщения (вариант UX — согласовать с юристом). |

## 3. Требования к продукту (акцепт и версии)

| Требование | Описание |
|------------|----------|
| **Явный акцепт** | При регистрации: чекбоксы на ПС, Пользовательское соглашение, согласие на ПДн; при необходимости — отдельно чувствительные данные и дисклеймер «не медицина». |
| **Нельзя обойти** | Регистрация без подтверждения ознакомления с ключевыми документами (механика — с юристом: один экран или пошагово). |
| **Версионирование** | У каждого документа на сайте: номер редакции и дата вступления в силу; при смене редакции — уведомление и журнал изменений (`Документы.md`). |
| **Обратная связь** | Заметная форма/канал для запросов по ПДн и жалоб (согласовано с `Документы.md` и `Безопасность.md`). |

## 4. Карточка «фактов для заполнения» (заполняет product до передачи юристу)

Юрист не придумает за нас содержание продукта. В черновики v0 нужно вставить **актуальные** данные:

- [ ] Полное наименование оператора, ИНН, ОГРН/ОГРНИП, юридический и почтовый адрес.
- [ ] Контакт для запросов по ПДн (e-mail; при назначении ответственного лица — как решит юрист).
- [ ] Домены и наименование сервиса для пользователя («Телотрон» и т.д.).
- [ ] Перечень обрабатываемых данных **по факту полей в приложении** на дату пилота.
- [ ] Перечень субпроцессоров: хостинг (провайдер, регион), БД, почта, аналитика, Sentry, иные API.
- [ ] Есть ли **менеджер** в пилоте и какие данные он видит (матрица — см. `Безопасность.md`).
- [ ] Пилот **платный или бесплатный** для тренера/клиента (влияет на необходимость оферты на оплату в v0).

---

## 5. Что в v0 намеренно не обязательно (если не заявлено иначе)

- Публичная **оферта на оплату** подписки, чеки, возвраты через платёжного провайдера — **после** появления приёма платежей в продукте (отдельный юридический подэтап).
- Полный **шаблон договора тренер–клиент** — желателен к пилоту как рекомендация, но может быть **вне** минимального v0, если тренеры используют свои договоры; уточнить с юристом (`юридические риски.md` §3).
- **Правила менеджеров и атрибуции** — по `Документы.md` ближе к первой волне продаж (до 01.07), не блокируют закрытый пилот без менеджеров.
