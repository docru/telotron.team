# Требования MVP v1 (продукт)

Документ отражает **что** нужно пользователю и продукту и **как проверить** готовность, без привязки к стеку и реализации. Технические спецификации — в папке [`../ТЗ/README.md`](../ТЗ/README.md).

## 1. Цель MVP
Запустить рабочую первую версию платформы, где специалист ведёт клиентов, назначает занятия и планы, клиент получает напоминания и отмечает базовый трекер, а команда получает обратную связь и базовую атрибуцию по партнёрским ссылкам.

## 2. Зафиксированные рамки
- Модель: один специалист = одна практика.
- Продажи на первом этапе силами основателя.
- Рост после старта через партнёрку.
- Отдельная поддержка как роль появляется после 500-1000 платящих.
- Тарифы: Light 0 / Pro 5000 / Max 10000.
- Промо: 2 месяца Max, затем Light.
- На Light ограничение по функциональности, не по числу клиентов.
- MAX канал: стратегический приоритет post-MVP (не блокер MVP).
- Пилот работает в режиме триала: в период пилота пользователям доступен весь функционал пилотной версии без тарифных ограничений.
- **В MVP нет встроенного чата** (ни личного «тренер–клиент», ни группового): коммуникации вне продукта или через уже используемые каналы (мессенджеры, звонки и т.д.).
- **В MVP нет роли менеджера** и отдельного клиентского приложения/кабинета менеджера; пользовательские роли в продукте — **клиент** и **специалист**, плюс **административная консоль** для команды (поддержка, аудит).

## 3. Авторизация и регистрация (M0)
- Основной логин: телефон.
- Email: опционально (если выбран пользователем как канал подтверждения/восстановления).
- После регистрации клиента обязательная короткая анкета: ФИО, пол, дата рождения.
- Каналы напоминаний в MVP:
  - по умолчанию in-app + Telegram;
  - при сценарии с email допускается дублирование на email.

Техническая спецификация единого сервиса авторизации (OTP, потоки регистрации/входа/восстановления): [тех-ТЗ-сервис-авторизации-MVP](../ТЗ/тех-ТЗ-сервис-авторизации-MVP.md).

Техническая спецификация фронтенда и PWA (стек, offline-first, структура модулей): [тех-ТЗ-frontend-и-PWA-MVP](../ТЗ/тех-ТЗ-frontend-и-PWA-MVP.md).

## 4. Состав MVP по модулям

### M0. Платформа и роли
Входит:
- Регистрация/вход по утверждённой схеме.
- Роли клиент/специалист.
- Базовый профиль пользователя.
- Подключение канала оповещений.

Не входит:
- Мультиролевая сложная оргмодель.
- Отдельные кабинеты для маркетолога/поддержки/руководителя.
- Роль менеджера и отдельный кабинет/приложение под неё (см. §2).

Критерии готовности:
- Пользователь проходит регистрацию и вход.
- Роль применяется корректно.
- Канал оповещений подключается и сохраняется.

### M1. Карточка клиента / CRM
Входит:
- Специалист отправляет клиенту приглашение по короткой постоянной ссылке.
- Клиент проходит по ссылке и появляется в кабинете специалиста.
- Механизм приглашения клиента и партнёрская ссылка работают на одном принципе.

Не входит:
- Ручное создание клиента специалистом.
- Статусы клиента (перенесено на следующий этап).

Критерии готовности:
- Специалист может поделиться ссылкой приглашения.
- Клиент после регистрации по ссылке автоматически привязывается к специалисту.
- Связь "клиент -> специалист" корректно сохраняется.

### M2. Календарь и журнал
Входит:
- Создание занятия: дата, время, клиент.
- Клиент видит предстоящие занятия.
- Специалист может перенести/отменить занятие.
- Нет критических конфликтов в базовом сценарии.

Не входит:
- Сложные календари и расширенные режимы.

Критерии готовности:
- Занятие создаётся, редактируется, отменяется.
- Клиент видит актуальное расписание.

### M4. Напоминания
Входит:
- Напоминание формируется для занятия.
- Отправка в утверждённые каналы MVP.
- Фиксация статуса доставки (успех/ошибка).
- Повторная отправка при временной ошибке.
- Тренер в календаре перед занятием видит статус "дошло/прочитано".

Не входит:
- Сложные сценарии автоматизации коммуникаций.

Критерии готовности:
- Напоминание уходит в канал.
- Статус доставки отображается.
- Статус прочтения виден тренеру в календаре.

### M5. Планы тренировок и питания
Входит:
- Специалист загружает план как файл (PDF/DOC/XLSX).
- Специалист назначает файл клиенту.
- Клиент может просмотреть и скачать файл.
- Сохраняются метаданные текущего назначения: кто назначил и дата.

Не входит:
- Встроенная библиотека готовых шаблонов.
- Конструктор планов.
- История версий планов.

Критерии готовности:
- Файл загружается и назначается клиенту.
- Клиент может открыть/скачать назначенный файл.

### M7-lite. Упрощённый трекер (дополнение MVP)
Входит:
- Тренировка: чекбокс.
- Приём пищи: текстовое поле.
- Шаги: числовое поле.
- Вода: числовое поле.

Не входит:
- Сон.
- Сложная аналитика, рекомендации, прогнозы.

Критерии готовности:
- Клиент может вносить и сохранять значения по всем четырём полям.

### M13. Динамика прогресса
Входит:
- Метрика по умолчанию: вес.
- Ввод данных только клиентом.
- История значений списком.
- Простой график динамики.

Не входит:
- Сложная аналитика, прогнозы, рекомендации.
- Фото-фиксация замеров.

Критерии готовности:
- Клиент вносит вес, данные сохраняются.
- И клиент, и специалист видят историю и график.

### M14. Обратная связь
Входит:
- Форма в приложении клиента и кабинете специалиста.
- Типы: ошибка / неудобно / идея.
- Обязательный текст.
- Автоконтекст: роль, версия приложения, время, id пользователя.
- Подтверждение "принято" после отправки.
- Единый канал обработки для команды.

Не входит:
- Тикет-система.
- Внутренняя переписка по обращению.
- Расширенный SLA-кабинет.

Критерии готовности:
- Форма отправляется с контекстом.
- Команда получает обращение.

## 5. Группы в MVP
Встроенный чат в продукте **отсутствует** в MVP целиком (см. §2); ниже «групповой чат» — только как запрет на будущее расширение сценария групп.

Входит облегчённый сценарий:
- В календаре есть режим/тип группового занятия.
- К одному занятию можно привязать несколько клиентов.
- Базовая отметка посещения/статуса.

Не входит:
- Групповой чат.
- Отдельные групповые тарифы/оплаты.
- Расширенная аналитика по группам.

## 6. Партнёрка
В MVP (минимум):
- С первого дня фиксируется атрибуция: кто подключился по чьей ссылке.
- Реферальные ссылки постоянные и пригодны для внешнего размещения.

Post-MVP:
- Начисления, балансы, выплаты.
- Расширенные партнёрские экраны и отчётность.

## 7. 152-ФЗ и ПД
Обязательные требования в MVP:

**Уведомление Роскомнадзора (gate до реальных пользователей)**
- До регистрации тренеров и клиентов в prod (период I / пилот на ПДн) — **подано уведомление** об обработке персональных данных в качестве оператора; критерии — [_telotron.ru/docs/Бизнес-требования/04-комплаенс-и-аудит/уведомление-роскомнадзор-оператор-пд.md](../../../_telotron.ru/docs/Бизнес-требования/04-комплаенс-и-аудит/уведомление-роскомнадзор-оператор-пд.md). Техническая готовность MVP **не заменяет** этот шаг.

**Акцепт и учёт согласий**
- Юрист готовит тексты; в продукте пользователь **не может** завершить регистрацию (и иные обязательные шаги первого входа, определённые юристом) без **явного** акцепта ключевых документов: отдельные чекбоксы, **без** предустановленного «согласен».
- В системе фиксируется факт акцепта: какие документы или типы согласия приняты, **редакция или версия** текста, **время** события (и иные поля по требованию юриста) — для ответов субъекту ПДн и внутреннего контроля.
- **Отдельное согласие** на обработку данных, относящихся к специальным/чувствительным категориям в понимании юриста, — в объёме и в моменте, которые задаёт юрист (в том числе при подключении сценариев **M7-lite** и **M13**).

**Публикация документов и версии**
- Политика конфиденциальности, пользовательское соглашение и иные обязательные к показу тексты **опубликованы** по постоянным адресам (URL); на экране документа пользователю видны **номер редакции** и **дата вступления в силу**.
- Из интерфейса доступны переходы по ссылкам: как минимум **регистрация** и **настройки профиля**, плюс иные места по согласованию с юристом (например подвал).
- При смене редакции документов — **уведомление** пользователей и **журнал изменений** юридических документов (сроки, канал уведомления, хранение журнала — по юридическому пакету проекта и регламентам).

**Прочее**
- Реализуется механизм обезличивания данных клиента.
- Реализуется логирование действий, связанных с ПД.

## 8. Этапный план

Календарь мероприятий до конца года (укрупнённо): [План мероприятий до конца 2026](../../01-Директор/Инструкции/План%20мероприятий%20до%20конца%202026.md).

1. Этап 1 (MVP-ядро): M0 + M1 + M2 + M4 + M5 + M7-lite + M13 + M14.
2. Этап 2 (Пилот-триал): запуск пилота с полным доступом к функционалу пилотной версии; цель этапа — протестировать и подтвердить пользовательскую ценность. **Предусловие:** уведомление РКН (§7) + закрытый юридический пакет v0/v1.
3. Этап 3 (параллельно пилоту): разработка коммерческого контура:
   - приём оплат от тренеров;
   - тарифы и управление доступностью функционала по тарифам;
   - партнёрка (атрибуция уже в MVP, далее начисления/балансы/выплаты и отчётность).
4. Этап 4 (коммерческий запуск): включение тарифных ограничений, запуск платного контура и расширенной партнёрки.
5. Этап 5 (после стабилизации): MAX, встроенный чат, кабинет/роль менеджера (при необходимости), прочие расширенные роли и улучшения.

## 9. Критерий готовности MVP в целом
MVP считается готовым, если:
- клиент и специалист проходят рабочий сценарий от регистрации до первого занятия и назначения плана;
- клиент получает напоминание и может вносить базовые трекер-данные;
- отображается динамика веса;
- работает обратная связь;
- фиксируется атрибуция по партнёрской ссылке;
- юридический минимум по 152-ФЗ реализован (в т.ч. **уведомление РКН**, акцепт и учёт согласий, публикация документов с редакциями по §7).
