# MVP — минимизация фазы 1

> **Разморозка (май 2026):** срез **M5 для тренировок** (план-файл) и **M7-lite** (галка «тренировка») **заменены** каноном в `_telotron.ru/docs/`: [разморозка-mvp-программы-и-трекеры](../../../../_telotron.ru/docs/Бизнес-требования/00-канон-mvp/разморозка-mvp-программы-и-трекеры.md), [программы-тренировок](../../../../_telotron.ru/docs/Бизнес-требования/02-модули/m5-programmy-trenirovok/программы-тренировок.md). Ниже — **исторический** минимальный срез; при противоречии побеждает канон в `docs/`.

Продолжение: `Этапность MVP — модули по приложениям.md`. Здесь зафиксирован **минимальный срез** модулей **M0, M1, M2, M4, M5, M13, M14** для первого релиза (два **клиентских модуля**: **№1 клиент**, **№2 специалист**). Модули **№3 (маркетолог)** и **№4 (руководитель)** отсутствуют в фазе 1.

Цель MVP (без расширений): специалист ведёт **карточки клиентов** и **расписание занятий**, назначает клиенту **готовый** план (без конструктора M6), клиент **видит** назначение и получает **напоминание о встрече**, есть **одна метрика прогресса** и **обратная связь**.

---

## Соглашения по формулировкам

| Термин | Значение |
|--------|----------|
| **В MVP** | Делаем в фазе 1 |
| **Вне MVP (фаза 1)** | Не делаем; либо фаза 2, либо ручной обход (таблица, звонок, внешний документ) |
| **Задел данных** | Поле/сущность в API или «тихая» запись без UI, чтобы не ломать модель потом |

---

## M0 — Платформа, авторизация, роли

| В MVP | Вне MVP (фаза 1) |
|-------|------------------|
| Регистрация/вход (email+пароль или выбранный тобой один способ OAuth — **один** вариант, не три) | Соцсети как единственный вход, SSO, magic link для всех ролей сразу |
| Роли: клиент, специалист, маркетолог (без отдельного UI в MVP); флаг/роль «руководитель» **без модуля №4** | Сложная матрица прав по полям, делегирование, кастомные роли |
| Одна организация (компания) на пилот **или** «специалист = своя мини-организация» — зафиксировать одну модель | Мультитенант с самообслуживанием биллинга организаций |
| API под три клиентских модуля; окружения dev/stage/prod по минимуму | Распределённые сервисы, очереди — только если уже не обойтись |
| Профиль пользователя: имя, контакт (телефон/email), привязка к роли | Богатый профиль, аватар обязателен, настройки темы (это ближе к M12) |

**Решение, которое нужно принять до кода:** один специалист = одна сущность «практика», или сразу «компания + специалисты внутри». От этого зависит расширяемость M1 и будущий контур №3.

## M1 — CRM и карточка клиента

### Приложение №2 (специалист)

| В MVP | Вне MVP (фаза 1) |
|-------|------------------|
| Список **моих** клиентов (ФИО, контакт, статус: активен / на паузе / завершил) | Воронка продаж, сложные стадии, автоматизации |
| Карточка клиента: контакты, **одно** текстовое поле «заметки» (или лента из **ручных** записей без шаблонов) | История звонков, интеграция с телефонией, теги из десятков типов |
| Привязка клиента к **одному** специалисту на пилоте | Передача клиента между специалистами, команда, права коллег |
| Создание клиента вручную + **одна** операция «пригласить клиента» (ссылка/код) для входа в №1 | Импорт из Excel на пилоте — по желанию, не как обязательный MVP |

### Приложение №3 (маркетолог)

| В MVP | Вне MVP (фаза 1) |
|-------|------------------|
| Не реализуется как отдельный клиентский модуль | Любые экраны лидов, воронки, статусов и отдельные панели роста |

**Примечание:** на фазе 1 продажи ведутся силами основателя, а рост обеспечивается партнёркой без отдельного интерфейса маркетолога.

## M2 — Журнал и календарь занятий

### Только клиентский модуль №2 (и отображение клиенту в №1 — см. ниже)

| В MVP | Вне MVP (фаза 1) |
|-------|------------------|
| Вид **неделя** (достаточно для старта); создание слота: дата, время начала/конца, клиент из списка M1 | Месяц с drag-and-drop, повторяющиеся серии, ресурсы/кабинеты |
| Тип события на пилоте **один** (например «онлайн-встреча») | Типы: очно/группа/консультация с разной длительностью |
| Отмена/перенос вручную | Автоправила конфликтов, ожидание очереди |
| Клиент в №1 видит **список своих предстоящих встреч** (дата, время, с кем) — данные из того же API | Полноценный общий календарь в №1 с подпиской в .ics на пилоте |

**Минимизация:** без перетаскивания и без повторов — сильно ускоряет фазу 1.

## M4 — Напоминания

| В MVP | Вне MVP (фаза 1) |
|-------|------------------|
| Напоминание клиенту **о предстоящей встрече** (за N часов, **одно** фиксированное N, настраивается глобально для пилота, не per-user) | Напоминания по трекерам (это связка с M7, фаза 2) |
| Канал доставки **один**: email **или** in-app **или** один мессенджер — выбрать один для MVP | Пуш на все платформы + несколько платных каналов доставки |
| Напоминание специалисту (опционально, простое): «завтра N встреч» | Сложные правила эскалаций |

**Минимизация:** in-app + email часто проще, чем пуши и модерация мессенджеров (M15).

---

## M5 — Планы тренировок и питания (без M6)

**Superseded (тренировки):** конструктор программ — см. канон выше. **Питание** в MVP по-прежнему **файл**.

### Приложение №2

| В MVP | Вне MVP (фаза 1) |
|-------|------------------|
| Библиотека **готовых** планов: загрузка админом/разработчиком (JSON/Markdown/таблица) **или** ручное создание записей в БД без UI редактора | Визуальный редактор планов, версионирование для клиента, сравнение версий |
| Действие «назначить клиенту план X с даты Y»; **один активный** план на тип (тренировки / питание) или явно «один план на клиента» на пилоте | Несколько параллельных программ, ветвление по неделям |
| Рецепты/КБЖУ: **статические** записи в библиотеке (как справочник), без автопересчёта сложных цепочек | Авторасчёт тоннажа, ССЗ, сложные формулы (M10) |
| Статьи и микроконтент | Вне MVP: отдельный CMS; на пилоте — ссылка на внешний документ в заметке M1 |

### Приложение №1

| В MVP | Вне MVP (фаза 1) |
|-------|------------------|
| Экран «Мой план»: текст/блоки **только чтение** + дата назначения | Отметка «выполнил упражнение» внутри плана (это уже задел M7) |
| Список рецептов **только просмотр**, если включено в пилот | Фото еды, дневник приёма пищи |

**Минимизация:** план = документ, не интерактивный конструктор.

## M13 — Динамика и графики

| В MVP | Вне MVP (фаза 1) |
|-------|------------------|
| **Одна** метрика на весь пилот (рекомендация по умолчанию: **вес** или **один замер** — обхват/жир %, что выберешь) | Набор метрик: сон, вода, шаги, самочувствие, множество графиков |
| Ввод: **специалист** фиксирует значение на приёме **или** клиент вводит **одно число** раз в период (неделя) | Полный M7: дневные логи, отклонения от плана, фото |
| Отображение: **простая таблица** «дата — значение» + **один** линейный график | Сравнение с планом, коридоры, экспорт |

**Задел данных (опционально в MVP):** хранить тип метрики в БД как строку/enum, чтобы потом добавить вторую метрику без миграции «ломающей» клиентов.

**Где UI:** в №2 — график по клиенту; в №1 — тот же график в упрощённом виде (или только последнее значение + мини-график).

---

## M14 — Обратная связь

| В MVP | Вне MVP (фаза 1) |
|-------|------------------|
| Одна форма: текст + тип (идея / неудобство / ошибка) + автоматический контекст (роль, версия клиента) | Скриншоты, голос, тикет-система, публичный roadmap |
| Доставка: email владельцу продукта **или** запись в таблицу/БД с простым просмотром в №2 | Интеграция с Jira/Linear |

## Сводная таблица по модулям фазы 1

| Модуль | Суть минимизации |
|--------|------------------|
| **M0** | Один способ входа, простая модель организации, два клиентских модуля к одному API |
| **M1** | Короткий список статусов; одно поле заметок |
| **M2** | Неделя, один тип события, без DnD и повторов |
| **M4** | Только встреча, один канал, фиксированное N |
| **M5** | Готовые планы извне; назначение «как документ»; без статей/CMS |
| **M13** | Одна метрика, таблица + один график |
| **M14** | Текстовая форма + маршрут уведомления |

---

## Что явно не обещаем в фазе 1 (чтобы не раздувать скоуп)

- **M3** чат, **M7** полноценный трекинг, **M8** видео, **M9** оплаты, **M15** интеграции — по `Этапность MVP` это фаза 2 (оплата может остаться вне системы до M9).
- **Приложение №4** — отсутствует.
- **M6** конструкторы — отсутствуют; контент готовится параллельно методистом/вручную.

---

## Проверка «MVP закрыт» (чек-лист)

- [ ] Специалист (№2): создал клиента, завёл встречи на неделю, назначил готовый план.  
- [ ] Клиент (№1): вошёл, видит встречи и план, получил напоминание о встрече.  
- [ ] Клиент или специалист: внесли значение **одной** метрики; оба видят динамику (таблица/график).  
- [ ] Любая роль: отправила обратную связь через M14.

После этого можно отдельной сессией пройтись по **оценке сроков** только для строк «В MVP» (см. `Функционал — модули и оценка.md`) и вынести №3 в отдельный этап роста после подтверждения потребности.
