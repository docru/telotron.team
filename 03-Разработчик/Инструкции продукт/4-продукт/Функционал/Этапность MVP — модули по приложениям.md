# Этапность MVP: четыре клиентских модуля и модули (M0–M16)

Связанные документы: `Функционал.md`, `Функционал — модули и оценка.md`. Термины «платформа» и «приложение»: [Глоссарий](../../../../01-Директор/Инструкции/1-контекст-и-правила/Глоссарий.md).

**Архитектура:** четыре **клиентских модуля** (отдельные сборки под роли; типично — **мобильное приложение** из стора, при необходимости — **веб-клиент** той же роли) + общая **платформа** на сервере (API, данные, интеграции). Нумерация ниже — как в решении «четыре клиента на одной платформе».

| № | Кто | Содержание |
|---|-----|------------|
| **1** | Клиент | Личный кабинет клиента |
| **2** | Специалист | Рабочее место тренера/диетолога |
| **3** | Маркетолог | Внутренний контур партнёрки и роста (не входит в MVP фазы 1) |
| **4** | Руководитель | Показатели компании |

**Важно по этапу 1:** клиентский модуль **№4 не делаем** — позже, на основе уже накапливаемых данных (агрегации, отчёты, дашборды). На этапах 1–2 закладываем **сбор событий и сущностей** так, чтобы потом не переписывать модель данных под руководителя.

**Нумерация модулей** — как в `Функционал — модули и оценка.md` (M0–M16).

---

## 1. Общая платформа (не четвёртый клиентский модуль из четырёх)

Одна кодовая база бэкенда (или чётко разделённые сервисы, но единая доменная модель): пользователи, организации/специалисты, клиенты, права доступа, журнал аудита при необходимости.

| Модуль | Назначение |
|--------|------------|
| **M0** | Авторизация, роли (клиент / специалист / маркетолог / поддержка(после 500-1000 платящих) / задел под руководителя), API, деплой |

На этапе 1 в M0 закладывается роль **«руководитель»** в модели прав **без UI** (или флаг «зарезервировано»), чтобы не ломать схему при появлении клиентского модуля №4.

---

## 2. Сводка: Фаза 1 / Фаза 2 / Отложено

Условные названия:

- **Фаза 1** — MVP: специалист ведёт клиентов и занятия, клиент видит назначенное и отмечает минимум. Отдельных продажников и менеджеров в продукте нет.
- **Фаза 2** — коммуникации, трекинг, оплаты, интеграции; старт внутренних инструментов маркетолога (партнёрка/рост) по приоритету.
- **Отложено** — не блокирует ценность MVP и сильно раздувает сроки; часть станет проще после накопления данных (в т.ч. под №4).

| Модуль | Кратко | Фаза |
|--------|--------|------|
| **M0** | Платформа, auth, RBAC, API под три клиентских модуля (№1–3) | **1** |
| **M1** | CRM, карточка клиента, заметки | **1** (MVP-срез) |
| **M2** | Журнал / календарь занятий | **1** (MVP-срез) |
| **M3** | Чат (+ привязка к планам/упражнениям) | **2** |
| **M4** | Напоминания (встречи, трекеры), дубли в мессенджеры | **1** (минимум) + **2** (расширение каналов) |
| **M5** | Планы тренировок и питания, рецепты, статьи | **1** (назначение + просмотр; контент из шаблонов) + **2** (редактор богаче) |
| **M6** | Конструкторы шаблонов питания и тренировок | **Отложено** (после MVP; до этого — внешние/ручные шаблоны) |
| **M7** | Трекеры, отклонения, фото еды, мотиваторы | **2** (нарастающий срез; в фазе 1 — только задел под данные, см. ниже) |
| **M8** | Видео тренировок | **2** (часто начинают с embed-ссылок) |
| **M9** | Оплаты, «Мой налог», удержания | **2** |
| **M10** | Удержание/привлечение, бизнес-метрики, акции, авторасчёты КБЖУ/ССЗ и т.д. | **Отложено** (правила/дашборды — возможны точечно во **2** как «простые отчёты», без «умных» рекомендаций) |
| **M11** | Группы (оплаты, оповещения, чат) | **Отложено** |
| **M12** | Персонализация (фото, цвета) | **2** (или отложено, если не критично для пилота) |
| **M13** | Графики прогресса | **1** (минимально: 1–2 метрики) + **2** (полнота) |
| **M14** | Обратная связь | **1** |
| **M15** | Интеграции (ЮKassa, ФНС, мессенджеры, пуш, маркетинговые сервисы) | **2** по мере необходимости; **Отложено** — узкоспециальные сервисы без ясного API |
| **M16** | Панели маркетолога/поддержки/руководителя | **Разделено:** см. раздел 4 |

**Шаблоны контента** (рецепты, наборы питания, упражнения, планы) — параллельный трек; в **фазе 1** не блокируют код, если планы вносятся «вручную»/импортом без M6.

|------|----------------------|

| Фаза | Модули / возможности |
|------|----------------------|

| **Позже** | Фактически новый фронт + агрегаты по уже собранным данным (бывший **M16** для руководителя выделяется сюда) |

## 4. Уточнение по M16 (было «панели продажника / менеджера / руководителя»)

После разделения на четыре клиентских модуля:

- **M16** в документе с оценками разбивается на:
  - функциональность **модуля №3 (маркетолог)** (начиная с фазы 2);
  - функциональность **поддержки** (после порога 500-1000 платящих);
  - функциональность **модуля №4** (позже, из данных).

В этой этапности **M16 не отдельным модулем в фазе 1**, а отложенный контур для маркетолога/поддержки и **задел данных** под №4.

---

## 5. Фаза 1 (MVP): что должно работать в целом

Критерий «MVP закрыт»: специалист ведёт календарь и карточки клиентов, назначает **готовые** планы/материалы, клиент их видит и получает напоминания о встрече, есть обратная связь и минимальная динамика (хотя бы одна метрика + график/таблица). Отдельный модуль маркетолога/поддержки в MVP не обязателен.

**Входит (сводно):** M0, M1 (срез), M2 (срез), M4 (минимум), M5 (без M6), M13 (минимум), M14 + **два клиентских модуля** (№1, 2).

**Намеренно не входит:** №3, №4, M6, M9 (если оплата на пилоте вне системы — иначе M9 во **2**), M3, богатый M7/M8/M10/M11/M15.

---

## 6. Фаза 2: логический порядок (без жёстких сроков здесь)

Рекомендуемый порядок внедрения модулей (можно частично параллелить только с осторожностью по одному разработчику):

1. **M3** (чат) — сильно повышает удержание, но дороже в поддержке.  
2. **M7** расширение + **M4** напоминания по трекерам.  
3. **M8** (embed → при необходимости загрузка).  
4. **M9** + релевантный **M15** (платежи, при необходимости ФНС).  
5. **M15** мессенджеры / пуш по выбранной стратегии.  
6. **M12**, углубление **M5**, расширение **M13**, запуск **№3 (маркетолог)** при подтверждённой необходимости.

---

## 7. Отложено (после фазы 2 или отдельные инициативы)

- **M6** — конструкторы шаблонов (до этого шаблоны готовятся вне продукта или простым импортом).  
- **M11** — группы.  
- **M10** — рекомендации по удержанию/привлечению, акции, маркетинговые «сервисы привлечения», сложные авторасчёты — как правило, после стабильных данных и процессов.  
- **Клиентский модуль №4** — дашборд руководителя на агрегатах.  
- Из **M15:** сервисы коллажа и прочие узкие интеграции без бизнес-кейса.

## 8. Следующий шаг (как ты описал)

Пройтись **только по модулям фазы 1** и для каждого зафиксировать **минимальный срез** (что выкидываем из M1, M2, M4, M5, M13 в первом релизе и что оставляем в **клиентских модулях** №1–3). Этот файл при этом можно не менять — добавить рядом, например, `MVP — минимизация фазы 1.md`.
