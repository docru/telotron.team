# invite_tokens: PK (ULID) и публичный токен ссылки

**Статус:** зафиксировано для реализации (обсуждение 2026-05-11).  
**Полный Q&A по зрелой системе ссылок** (вопросы и ответы по мере согласования): [invite-links-voprosy-i-otvety.md](invite-links-voprosy-i-otvety.md).  
**Контекст:** партнёрские и пригласительные ссылки (`invite_tokens`); ориентир — копипаст и пересылка, не диктовка.

## Решение

1. **Первичный ключ** таблицы `invite_tokens` — **ULID** (не автоинкремент `bigint`). Используется **внутри системы**: FK из других таблиц, аудит, админка «по id сущности», внутренние API при необходимости.

2. **Отдельное поле `token`** — строка, которая попадает в **публичный URL** (короткий/«красивый» сегмент по выбранным правилам генерации). По нему ищем запись на лендинге и при приёме инвайта. **Не считаем** отдельный «второй секрет» ради самого факта публичной ссылки: код намеренно расшаривается; защита — продуктовая (вход, правила приёма) и антиабьюз (лимиты и т.д.).

3. **Миграция данных не требуется** на момент принятия: пустая база — достаточно **миграции схемы** и переделки кода.

## Зачем два идентификатора

- **ULID** — стабильный внутренний суррогат без предсказуемых bigint, удобно для связей и логов.
- **`token`** — то, что тренер/платформа отдают наружу; длина и алфавит подстраиваются под UX (копипаст), не смешиваясь с внутренним PK.

## Не делаем

- Отдельное поле «секрет» **только потому**, что токен лежит в БД: для сценария «ссылка = приглашение» это избыточно, если нет отдельного требования к двум уровням идентификаторов в URL vs API.
