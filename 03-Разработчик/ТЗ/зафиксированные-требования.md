# Зафиксированные требования

Документ фиксирует уже принятые требования от бизнеса, юриста и продуктовых решений, без детализации реализации.

## Статус и роль документа

- Статус: рабочий канон для разработки в ветке `Команда/03-Разработчик/ТЗ`.
- Роль: источник «что нужно продукту», а не «как это реализовать».
- Изменения вносятся только после явного решения заказчика/владельца продукта.

## 1) Границы продукта и зоны

- Зоны: `Pro`, `Client`, `Admin`.
- Для `Admin` отдельного PWA-приложения нет.
- `Admin` работает как web-панель (Filament) во внутреннем контуре.
- Авторизация `Admin`: телефон + пароль, доступ только из приватной сети.

## 2) Авторизация и сессии

- Для `Pro/Client` пароли не используются.
- Контур входа: `Passkey + OTP`.
- Сессия запоминается, чтобы не запрашивать OTP на каждое действие.
- API/PWA в same-origin, серверная сессия (Sanctum SPA) принята как базовый вариант.

## 3) Ключевые решения по MVP-модулям

- `M1`: инвайты/клиентская база, клиентские и партнёрские ссылки, авто-создание ссылок тренеру.
- `M2`: календарь/запись на слот, обработка конкуренции записи, понятные сообщения пользователю.
- `M4`: подтверждение участия через каналы, in-app обязателен, Telegram опционален.
- `M5`: планы тренировок, в MVP локальное хранилище с API-выдачей; переход к object storage вынесен post-MVP.
- `M7-lite`, `M13`, `M14`: согласованы интерфейсные границы MVP без расширенных аналитик.

## 4) Юридический и аудиторский минимум MVP

- Ведётся аудит критичных действий и согласованных read-событий.
- Минимизация ПД в логах обязательна.
- Срок хранения аудит-данных зафиксирован в профильных документах.
- Для запроса на удаление ПД в MVP используется указание контактного email в профиле (без отдельного in-app workflow).

## 5) Решения по API-контракту

- Формат ошибок: RFC 9457 (problem details) с ограничением на утечку чувствительных данных.
- Идемпотентность мутаций: `idempotency_key` в теле запроса.
- Для конкурентной записи на занятие принята модель транзакции с блокировками и `409 Conflict` при отсутствии слота.

## 6) Версии и релизы

- Версионирование ведётся через `config/version.php`.
- Используется общий `version` и зональные `build.pro|client|manager`.
- Build увеличивается только в затронутой зоне.
