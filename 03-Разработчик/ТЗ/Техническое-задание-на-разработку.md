# Техническое задание на разработку (единый канон MVP)

## 1. Назначение документа

Этот документ фиксирует полный согласованный набор технических требований для разработки MVP Telotron.

Правила применения:

- Это единый источник истины для команды разработки по MVP.
- Если требование не описано здесь, оно не считается обязательным к реализации в MVP без отдельного согласования.
- При конфликте с любыми другими материалами приоритет у данного документа.

## 2. Границы MVP и зоны продукта

### 2.1 Зоны

- `Pro` — PWA-приложение для тренера.
- `Client` — PWA-приложение для клиента.
- `Admin` — web-панель на Laravel Filament.

### 2.2 Ключевое ограничение по Admin

- Для `Admin` отдельного PWA-приложения нет.
- `Admin` доступен только внутри приватной сети.
- Вход в `Admin`: телефон + пароль.

## 3. Технологическая архитектура

### 3.1 Backend

- Архитектурный подход: модульный монолит Laravel.
- Структура модулей: `app/Modules/<Модуль>/`.
- Реализация через Eloquent и сервисный слой приложения.
- На этапе MVP не требуется обязательное внедрение VO/Entity/Repository-паттернов в стиле строгого DDD.

### 3.2 Frontend

- Стек: Vue 3 + TypeScript + Pinia + Tailwind.
- Две отдельные клиентские зоны в одном репозитории:
  - `Pro` (PWA),
  - `Client` (PWA).
- Общие UI/доменные блоки используются совместно между зонами там, где это уместно.
- Подход: offline-first для PWA-зон.

## 4. Авторизация, сессии и доступ

### 4.1 Pro/Client

- Пароли не используются.
- Контур входа: `Passkey + OTP`.
- Сессия запоминается, чтобы не запрашивать OTP для каждого ежедневного действия.
- API и PWA работают в same-origin контуре.
- Сессионная модель: серверная сессия (Sanctum SPA).
- На холодном старте не требуется принудительная мутация; запрос CSRF выполняется перед первой мутацией.

### 4.2 Admin

- Отдельные учётные записи админов.
- Вход: телефон + пароль.
- Доступ ограничен приватной сетью.

## 5. HTTP/API контракт

### 5.1 Формат ошибок

- Используется RFC 9457 (Problem Details for HTTP APIs).
- Ошибки должны быть безопасными: без утечки чувствительных данных.
- Для бизнес-конфликтов используется корректный HTTP-статус (например, `409 Conflict`).

### 5.2 Идемпотентность мутаций

- Для `POST` / `PATCH` / `PUT` мутаций используется `idempotency_key` в JSON body.
- Повтор с тем же ключом и тем же телом должен возвращать прежний итог операции.
- Повтор с тем же ключом и другим телом должен возвращать конфликт.
- Для multipart-запросов идемпотентность также поддерживается через `idempotency_key`.

### 5.3 Конкурентная запись на слот (M2)

- Для исключения гонки записи используется транзакционная блокировка на БД (`DB::transaction` + `SELECT ... FOR UPDATE`).
- При отсутствии мест возвращается `409 Conflict` в формате RFC 9457.
- При идемпотентном повторе результат должен быть консистентным (успех/конфликт как в первичном результате).

## 6. Модульные требования MVP

## M0. Базовые профили и настройки

### Pro/Client

- В профиле присутствует управление passkey.
- В профиле размещается контактный email для подачи запроса на удаление персональных данных.
- В MVP отдельный продуктовый workflow ПДн-заявок не реализуется.

### Admin

- Базовое управление данными в web-панели.

## M1. Инвайты и клиентская база

### Pro — список клиентов и карточка клиента

- Список клиентов и карточка клиента относятся к M1 (не к M0).
- Над списком клиентов есть кнопка «Пригласить нового клиента».
- Варианты действия при приглашении:
  - строка ссылки с копированием по клику,
  - системный share через кнопку «Поделиться».

### Pro — клиентские и партнёрские ссылки

- Ссылки редактируются на странице настроек.
- На тарифе `Maximum` и на пробном периоде допустимо множество партнёрских ссылок.
- После окончания trial на `Lite/Profi`:
  - все ранее созданные ссылки отображаются,
  - редактирование недоступно (read-only),
  - удаление доступно как soft delete.
- Для тренера должен сохраняться минимум один активный клиентский onboarding-link.
- При создании нового тренера автоматически создаются:
  - одна клиентская ссылка,
  - одна партнёрская ссылка.

### M1 аудит действий по ссылкам

- Логируются критичные события:
  - `created`,
  - `revoked`,
  - `deleted`,
  - `restored`.
- Изменение только title ссылки в MVP отдельным аудит-событием не логируется.

## Модуль «Группы»

### Pro

- Экран «Список групп»:
  - есть кнопка «Добавить группу»,
  - отображаются все группы,
  - в MVP без пагинации, фильтров и сортировок.
- Экран «Группа»:
  - отображается название группы,
  - кнопка «Добавить клиента в группу»,
  - попап с эластичным фильтром по имени/фамилии,
  - список клиентов для добавления.

### Client/Admin

- Функциональность согласована без дополнительных усложнений относительно MVP.

## M2. Календарь и журнал занятий

### Pro

- При создании занятия из календаря открывается pop-up редактирования.
- В pop-up задаются:
  - дата/время,
  - тип занятия (индивидуальное/групповое),
  - группа (для группового сценария).
- При клике на существующее занятие открывается pop-up с:
  - типом,
  - составом участников,
  - возможностью отмены занятия с обязательным указанием причины.

### Client

- Календарь обновляется раз в минуту.
- В офлайне слоты визуально тусклее и некликабельны.
- Над календарём отображается индикатор offline.
- При попытке записи в офлайне показывается вежливое понятное сообщение с рекомендацией проверить сеть/перейти в онлайн.
- При конфликте записи (место занято в момент отправки) показывается вежливое сообщение о невозможности записи.

### Перенос занятия и подтверждение участия

- При любом изменении `starts_at` и/или `ends_at`:
  - `attendance_confirmed_at` сбрасывается,
  - `attendance_confirmed_via` сбрасывается,
  - требуется новое подтверждение участия,
  - напоминания формируются заново.

## M4. Напоминания о занятиях

### Каналы

- In-app — обязательный минимальный канал информирования.
- Telegram — опциональный канал.

### Подтверждение участия

- Уведомления о занятии содержат действие подтверждения участия.
- Подтверждение в одном канале должно снимать запрос на подтверждение в остальных каналах.
- Для тренера должно быть видно:
  - кто подтвердил,
  - кто не подтвердил,
  - когда и через какой канал подтверждено.

### Напоминания после подтверждения

- После подтверждения участие считается подтверждённым.
- Информационные напоминания продолжают отправляться.
- Кнопка «Подтвердить» в последующих напоминаниях не показывается.

## M5. Планы

### Pro

- В настройках есть раздел «Планы»:
  - список планов,
  - добавление нового плана с названием.
- В карточке клиента есть кнопка «Планы», открывающая модал:
  - список назначенных планов,
  - добавление плана из библиотеки,
  - удаление назначения плана клиенту.

### Client

- На главной странице отображается список ссылок на назначенные планы для скачивания.

### Хранение и доступ к файлам в MVP

- Для MVP используется локальное файловое хранилище Laravel.
- Выдача файлов идёт через API с проверкой прав назначения.
- Удаление файла плана запрещено, если он назначен клиентам.
- При попытке удаления назначенного файла API возвращает конфликт с указанием затронутых клиентов и ссылок на их карточки.

### Масштабирование после MVP

- Переход к объектному хранилищу (S3-совместимому) и signed URLs рассматривается как post-MVP задача.

## M7-lite. Трекер дня

### Client

- На главной странице быстрые поля/чекбоксы ежедневного трекера.
- Критично быстрое открытие с готовностью к вводу данных.
- Доступен режим календарного просмотра прошлых дней в read-only.
- Скорость загрузки read-only календаря не является критическим NFR.

### Pro

- В карточке клиента на календаре отображаются пиктограммы метрик с числами.
- Графики по M7-lite в MVP не входят.

## M13. Динамика веса

### Client

- Поле веса на главной странице.
- По умолчанию отображается последний замер (или пусто, если замеров нет).
- Загрузка последнего веса не должна блокировать готовность страницы для ежедневного ввода.
- Кнопка «Прогресс» открывает модал с графиком веса:
  - дефолт: 3 месяца,
  - доступные периоды: 1/3/6/12 месяцев.

### Pro

- В карточке клиента вес отображается как кликабельный элемент.
- По клику открывается тот же модал прогресса веса.

## M14. Обратная связь

- Модуль принят без дополнительных изменений к согласованной версии MVP.

## 7. Схема данных и сущности (ключевые требования)

### 7.1 Инвайты и ссылки

- `invite_tokens` поддерживает:
  - несколько onboarding-ссылок при разрешённом тарифном сценарии,
  - title,
  - soft-delete (`deleted_at`).

### 7.2 Подтверждение участия

- В связке назначения клиента на занятие используются поля:
  - `attendance_confirmed_at`,
  - `attendance_confirmed_via`.
- Эти поля не заменяют бизнес-статус участия и используются именно для механики подтверждения выхода.

### 7.3 Пользователи и auth-ограничения

- Для `Pro/Client` парольная модель не используется в пользовательском потоке.
- Для `Admin` предусматривается отдельная модель учётных данных (телефон + пароль).

### 7.4 OTP

- Назначения OTP включают recovery-сценарии для passkey-контура.

### 7.5 Аудит

- Таблица аудит-событий хранит технический статус результата действия (`status_code`) и необходимый мета-контекст.
- Запрещено логировать секреты и полные чувствительные значения.
- Поддерживается срок хранения и автоматическая очистка в соответствии с юр-минимумом MVP.

## 8. Журналирование и ПДн

### 8.1 Write-аудит

- Логируются критичные действия пользователей, в том числе по ссылкам M1 и другим чувствительным операциям.

### 8.2 Read-аудит

- Логируются события просмотра списков и карточек сущностей согласно согласованному набору.
- Для list-view событий логируется одно событие на запрос/экран, а не событие на каждый элемент списка.
- В metadata включаются только безопасные поля (например, фильтры без ПД, диапазон дат, количество результатов, пагинация).

### 8.3 ПДн-запросы в MVP

- В MVP реализуется упрощённый подход:
  - в профиле указан контактный email для запроса удаления ПД.
- Отдельный in-app реестр `privacy_requests` и workflow обработки в продукте в MVP не входят.

## 9. Нефункциональные требования (NFR)

- Высокая отзывчивость главной страницы `Client` для ежедневных операций (трекер дня и вес).
- Понятное и вежливое UX-поведение в офлайне и при конфликте записи.
- Надёжная конкуррентная обработка записи на занятия.
- Безопасная обработка ошибок API без утечки чувствительных данных.
- Идемпотентность всех мутаций для защиты от повторных отправок и нестабильных сетей.

## 10. Политика версий и релизов

- Каноничный источник версии: `config/version.php`.
- Используются:
  - общий `version`,
  - зональные `build.pro`, `build.client`, `build.manager`.
- Build повышается только для зоны, где изменения должны доехать до установленного PWA (SW/precache/layout/offline-ресурсы).
- Build других зон не повышается «на всякий случай».

## 11. Чёткие ограничения MVP (что не делаем сейчас)

- Не делаем отдельное PWA для Admin.
- Не внедряем тяжёлую DDD-обвязку (обязательные VO/Entity/Repository) как требование MVP.
- Не внедряем в MVP отдельный продуктовый контур регистрации/контроля ПДн-запросов.
- Не внедряем object storage как обязательный runtime для M5 на старте.
- Не добавляем расширенные графики вне утверждённых границ M7-lite/M13.

## 12. Критерии готовности реализации по данному ТЗ

- Реализованы все требования модулей M1, M2, M4, M5, M7-lite, M13, M14 в границах, описанных выше.
- Реализованы принятые требования по auth/session для всех зон.
- API-контракт соответствует принятым правилам ошибок, идемпотентности и конкурентной записи.
- Аудит и логирование соответствуют юр-минимуму и требованиям минимизации ПД.
- Версионирование релизов зон реализовано через `version + build.<zone>`.

### 13.2 M1: инвайты и связи

| Поле | Тип/роль |
|------|----------|
| `id` | PK |

## 14. Полные интерфейсы MVP (утверждено)

### 14.1 Кросс-модульный Calendar

- Общий кастомный компонент календаря для M2/M7-lite.
- Режимы: `month`, `week`, `day`, `agenda`.
- Слоты: `day` (ячейка дня), `entry` (запись, варианты для week/day/agenda).
- Время: сервер UTC, отображение в TZ пользователя.
- Мутации расписания - только онлайн.
- Для Client при офлайне слоты тусклые и некликабельные, с явным offline-индикатором.
- Polling календаря при открытом экране: 1 раз в минуту (как база MVP).
- Drag-and-drop не обязателен в MVP.

### 14.2 M0 Интерфейсы

### Pro/Client
- Экран настроек: профиль, каналы уведомлений, управление passkey, выход.
- Раздел персональных данных: контактный email для запроса удаления ПД.

### Admin
- Только web-контур Filament.
- Отдельного PWA нет.
- Вход: телефон + пароль, приватная сеть.

### 14.3 M1 Интерфейсы

### Pro
- Экран списка клиентов:
  - кнопка «Пригласить нового клиента» над списком,
  - действия: копирование ссылки и системный share.
- Карточка клиента:
  - шапка с полом/ФИО/возрастом,
  - кликабельный вес с открытием модалки прогресса.
- Секция ссылок:
  - авто-созданные клиентская и партнёрская ссылки,
  - многоссылочность `client_onboarding` для Maximum/trial,
  - post-trial на Lite/Profi - read-only, только soft delete.

### Client
- Экран принятия инвайта и onboarding.

### Admin
- Read-only контур для расследований инвайтов/рефералов.

### 14.4 Интерфейсы модуля «Группы»

### Pro
- Экран «Список групп»:
  - кнопка «Добавить группу»,
  - вывод всех групп, без пагинации/фильтров/сортировки.
- Экран «Группа»:
  - название,
  - кнопка «Добавить клиента в группу»,
  - всплывающее окно с гибким поиском по имени/фамилии,
  - список участников.

### Client
- Отдельного каталога групп нет.

### Admin
- Read-only просмотр.

### 14.5 M2 Интерфейсы

### Pro
- При создании занятия: pop-up редактирования (дата/время, тип, группа).
- При клике на занятие: тип, состав, отмена с обязательной причиной.
- Перенос времени через форму редактирования.

### Client
- Обновление календаря раз в минуту.
- В онлайне слоты кликабельны, в офлайне нет.
- Сообщение при офлайне и отдельное сообщение при конфликте записи.

### Admin
- Read-only расписание для поддержки.

### 14.6 M4 Интерфейсы и каналы

### Pro
- На занятии отображаются:
  - количество подтвердивших,
  - количество не подтвердивших,
  - детальные списки в карточке занятия.

### Client
- Напоминания в in-app (обязательно) и Telegram (опционально).
- Кнопка «Подтвердить» в активном запросе подтверждения.
- После подтверждения:
  - запрос подтверждения снимается в других каналах,
  - дальнейшие напоминания остаются информирующими (без кнопки).

### Admin
- Контур просмотра логов доставок и диагностики.

### 14.7 M5 Интерфейсы

### Pro
- В настройках раздел «Планы»:
  - список,
  - добавление плана с названием.
- В карточке клиента кнопка «Планы»:
  - список назначенных,
  - добавление из библиотеки,
  - удаление назначения.
- При попытке удалить назначенный файл - показать список затронутых клиентов со ссылками на карточки.

### Client

### Admin
- Просмотр метаданных файлов по регламенту.

### 14.8 M7-lite Интерфейсы

### Client

- История собственных обращений.
- Входящие от поддержки и статус прочтения.

- Очередь обращений, карточка, ответы, проактивные сообщения.

### 14.11 Интерфейсы авторизации

### Pro/Client

  - фильтры и ограниченный доступ по роли.
