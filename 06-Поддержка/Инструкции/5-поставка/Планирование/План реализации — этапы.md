# Общий план реализации проекта (этапы и сферы)

Цель документа: не упустить важное в **подготовке** и **реализации** до пилота и первого платного контура.  
Сферы перечислены полностью; этапы сквозные — в каждом этапе есть задачи по релевантным сферам.

**Связанные артефакты в репозитории:**
- продукт: `Сырьё и исследования/Интервью/MVP/MVP-1.md`, `Сырьё и исследования/Интервью/Обобщение интервью/`;
- разработка: `Решения/6-реализация/Разработка/технические требования.md`;
- юридическое: `Решения/8-право-и-бренд/юридический раздел/юридические риски.md`;
- бизнес: `Сырьё и исследования/Бизнес-план/`.

**Принцип разработки** (из техтребований): сначала техдок → код → тесты; быстрый старт PWA с догрузкой фич; обновляемость PWA с сервера; надёжность; тесты; проверки безопасности при деплое.

---

## Сферы, которые нужно закрыть (чек-лист на весь проект)

| Сфера | Зачем нужна | Ключевые артефакты / действия |
|--------|-------------|-------------------------------|
| **Стратегия и позиционирование** | не расползтись по сегментам | ICP, гипотеза ценности, что **не** делаем в MVP-1 |
| **Продукт и исследования** | строить то, за что платят | интервью, сводка ЦА, MVP-1, критерии успеха пилота |
| **Юридическое и комплаенс** | запуск без «сюрпризов» | ПДн, оферта, согласия, дисклеймеры; см. `юридические риски.md` |
| **Монетизация и финансы** | устойчивость | модель цены, юнит-экономика пилота, сценарии оплат |
| **Дизайн и UX** | скорость внедрения у тренеров | сценарии, прототипы, гайдлайн, доступность |
| **Архитектура и разработка** | предсказуемый релиз | техдок, API, три мобильных приложения по ролям (и веб-клиенты), PWA, офлайн-стратегия |
| **Данные и безопасность** | доверие и 152-ФЗ | модель данных, роли, шифрование, бэкапы, журнал доступа |
| **Инфраструктура и DevOps** | стабильность | окружения, CI/CD, мониторинг, секреты |
| **Качество: тесты и приёмка** | регрессии не съедают время | пирамида тестов, чек-листы пилота, критерии «готово» |
| **Контент и методология** | непустой продукт в руках | шаблоны планов, тексты дисклеймеров для клиента |
| **Поддержка и обучение** | снижение отвалов пилота | FAQ, онбординг, канал обратной связи |
| **Маркетинг и продажи пилота** | добраться до реальных пользователей | лендинг/заявка, 3–5 пилотных тренеров, сбор метрик |
| **Операционка проекта** | сроки не срываются | дорожная карта, риски, решения по стеку, внешние подряды (юрист) |
| **Риски и зависимости** | заранее видеть блокеры | платёжка, ФНС, пуши iOS, хранение ПДн в РФ — по мере необходимости |

## Этап 0 — Зафиксировать рамки (1–3 недели календарно, параллельно с интервью)

**Цель:** одна страница «что строим и для кого», без кода.

| Сфера | Результат этапа |
|--------|-----------------|
| Стратегия | ICP для MVP-1, список **исключений** из сегмента (пример: клубный зарплатный тренер без своей клиентской базы) |
| Продукт | зафиксирован `MVP-1.md` или его версия 1.0; метрики успеха пилота |
| Юридическое | прочитан черновик рисков; решение: **когда** подключать юриста (до пилота обязательно) |
| Монетизация | гипотеза цены (фикс / за клиента) и что измеряем в пилоте |
| Операционка | владелец продукта, кто принимает решения; частота синков |

**Выход:** документ «рамки MVP-1» + согласованный список пилотных тренеров (кол-во, критерий отбора).

---

## Этап 1 — Продуктовая спецификация и юридический минимум (2–5 недель)

**Цель:** можно отдавать в разработку без «додумывания в коде».

| Сфера | Результат этапа |
|--------|-----------------|
| Продукт | user stories / сценарии по блокам MVP-1 (CRM, календарь, планы, отчётность, напоминания, чат, динамика) |
| Дизайн | wireframes ключевых экранов (тренер + клиент); согласованные состояния ошибок и пустых данных |
| Юридическое | пакет текстов v0: политика конфиденциальности, пользовательское соглашение, согласие на ПДн, дисклеймер «не медицина»; **ревью юриста** |
| Данные | модель сущностей (клиент, группа, план, отчёт, событие, роль); матрица доступа |
| Монетизация | сценарий пилота: платят ли тренеры символически / бесплатно / с депозитом; что обещаем |
| Риски | список «красных флагов» из интервью → требования (скорость, простота, не размножать отдельные клиенты без нужды) |

**Выход:** `ТЗ` или набор спецификаций в `Решения/6-реализация/Разработка/` + подписанные/согласованные юридические тексты для пилота.

## Этап 2 — Архитектура, стек, инфраструктура (1–3 недели, параллельно с дизайном)

**Цель:** стабильный каркас под три **клиентских модуля** (мобильные приложения по ролям и/или веб-клиенты) и общий API платформы.

| Сфера | Результат этапа |
|--------|-----------------|
| Архитектура | границы сервиса, auth/RBAC, версионирование API, стратегия PWA (обновление с сервера — из техтребований) |
| Инфраструктура | dev/stage/prod, секреты, CI, деплой, бэкапы БД |
| Безопасность | базовый threat model, HTTPS, rate limit, зависимости, план проверки при каждом деплое (как в техтребованиях) |
| Качество | стандарты: ветки, code review, минимальное покрытие критичных модулей тестами |

**Выход:** архитектурная схема + репозиторий с «скелетом» и первым деплоем пустой **платформы** (заготовка клиентов и API).

---

## Этап 3 — Реализация MVP-1 Core (ориентир: основной объём разработки)

**Цель:** тренер ведёт клиентов, календарь, планы, отчётность, напоминания — без «тяжёлых» интеграций.

Состав по `MVP-1.md`, разумное разбиение:

1. **Core:** CRM + календарь/запись + планы/материалы + отчётность + напоминания.  
2. **Расширение core:** динамика (графики) + продления/дни рождения в простом виде.

| Сфера | Результат этапа |
|--------|-----------------|
| Разработка | функциональность по спецификации; документация обновляется **до** кода |
| Качество | автотесты на доменную логику; ручные сценарии регресса |
| Контент | минимальный набор шаблонов планов для демо и пилота |
| Поддержка | черновик FAQ для пилотных тренеров |

**Выход:** задеплоенный stage, внутренний «приёмочный» прогон по чек-листу.

---

## Этап 4 — MVP-1 Plus (коммуникации и удержание в продукте)

**Цель:** закрыть самые болезненные разрывы из интервью — «всё в одном месте».

| Сфера | Результат этапа |
|--------|-----------------|
| Разработка | чат тренер–клиент; комментарии к группе/клиенту; сигналы «выпал из отчётности» |
| UX | полировка онбординга клиента (минимум трения) |
| Юридическое | правила чата, хранение переписки, сроки хранения — в документах v1 |

**Выход:** версия для полевого пилота на реальных клиентах тренера.

---

## Этап 5 — Пилот на 2–5 тренерах (4–8 недель наблюдения)

**Цель:** доказать ценность и собрать вход в платный контур.

| Сфера | Результат этапа |
|--------|-----------------|
| Маркетинг/пилот | отбор тренеров, договорённости, сбор обратной связи по фиксированному шаблону |
| Продукт | метрики из `MVP-1.md`; решение: что выкидываем/что переносим в MVP-2 |
| Поддержка | оперативные фиксы, канал связи, дневник инцидентов |
| Монетизация | решение по цене и модели после цифр пилота |
| Юридическое | при появлении оплат в продукте — отдельный подэтап с юристом (оферта, возвраты, чеки) |

**Выход:** отчёт пилота + обновлённый бэклог MVP-2 + готовность к коммерческому запуску или второму кругу пилота.

---

## Этап 6 — Коммерция и масштаб (после валидации)

**Цель:** предсказуемый рост без ломки архитектуры.

Типичные блоки (по мере зрелости):

- платежи и юридически выверенный биллинг;
- самозанятость/ИП/чеки — только после сценария с юристом;
- углублённый маркетинг внутри продукта (лиды, воронка) — если подтвердится спросом;
- отдельные интеграции (карты, мессенджеры, пуши) — по приоритету после ядра.

---

## Что не забыть «сквозняком» (на всех этапах)

1. **Документация раньше кода** — по твоему правилу из техтребований.  
2. **Безопасность при каждом деплое** — зависимости, конфиги, доступы.  
3. **Резервное копирование и восстановление** — до появления реальных клиентов.  
4. **Версионирование и откат релизов.**  
5. **Персональные данные** — минимизация сбора, сроки хранения, удаление по запросу (заложить в дизайн БД заранее).  
6. **Связь с MVP-1** — любая новая фича проходит проверку: «это нужно пилоту или это отвлечение?»

## Краткая дорожная карта «сверху вниз»

```
Этап 0: рамки + ICP + метрики
    ↓
Этап 1: ТЗ + юридический пакет v0 (юрист)
    ↓
Этап 2: архитектура + инфра + безопасность
    ↓
Этап 3: MVP-1 Core (разработка + тесты + контент)
    ↓
Этап 4: MVP-1 Plus (чат + удержание)
    ↓
Этап 5: пилот → отчёт → цена → MVP-2
    ↓
Этап 6: коммерция и масштаб
```
