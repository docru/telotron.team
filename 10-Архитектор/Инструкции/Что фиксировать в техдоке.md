# Что фиксировать в техдоке

Архитектор поддерживает **канон** в `_telotron.ru/docs/Техдок/` и журнал **ADR** (Architecture Decision Record).

## Где что лежит

| Тип | Путь |
|-----|------|
| Долгоживущий канон MVP | `Техдок/01-канон-mvp/` (backend, frontend, api-http, надёжность) |
| Зоны Pro/Client/Admin | `Техдок/02-зоны/` |
| Модульные детали | `Техдок/03-модули/` |
| ADR (решение + контекст + последствия) | `Техдок/00-мета/архитектурные-решения/` |
| План этапов | `Техдок/00-мета/план-реализации-mvp.md` |

Черновики и обсуждение до принятия — в `Команда/10-Архитектор/Исследования/` (не канон для кода).

## Когда писать ADR (обязательно)

1. Новый **модуль** или сущность, пересекающая 2+ существующих модуля.
2. Изменение **контракта API** (поля, коды, breaking change).
3. Новая **сквозная** capability: billing, partner payouts, push, MAX, native wrapper.
4. Смена **стратегии offline** / outbox / conflict resolution для модуля.
5. Отказ от ранее принятого решения («мы больше не делаем X»).

## Когда достаточно правки канона (без отдельного ADR)

- Уточнение имён папок, ссылок, примеров в уже принятом документе.
- Дополнение api-http одним endpoint в существующем §4.1x по шаблону.

## Шаблон ADR

Имя файла: `YYYY-MM-DD-краткое-имя.md`

```markdown
# ADR: [краткий заголовок]

**Статус:** proposed | accepted | superseded by …  
**Дата:** YYYY-MM-DD  
**Контекст:** …  
**Решение:** …  
**Альтернативы:** …  
**Последствия:** плюсы, минусы, что сделать в коде/docs  
**Связь:** ссылки на ТЗ, api-http, модули
```

После **accepted** — обновить канон (backend/frontend/api-http), если решение затрагивает их.

## Принципы документирования

- **Коротко:** ADR на 1–2 экрана, не роман.
- **Один источник истины:** код не противоречит accepted ADR; если противоречит — баг или срочный ADR supersede.
- **На русском** в docs; идентификаторы кода — латиница.
