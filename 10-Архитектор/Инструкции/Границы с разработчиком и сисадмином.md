# Границы с разработчиком и сисадмином

## Разработчик (03) vs архитектор (10)

| Тема | Архитектор | Разработчик |
|------|------------|-------------|
| Новый модуль M* | Чеклист, границы, ADR при спорных местах | Реализация, тесты, PR |
| API endpoint | Схема в api-http §4.1x, коды ошибок, идемпотентность | Контроллер, сервис, feature-тест |
| Рефакторинг «на будущее» без задачи | **Стоп** или ADR «отложено» | Только в scope задачи |
| Выбор библиотеки (крупная) | Предложение + trade-off в ADR | PoC, интеграция |
| Техдолг в одном файле | Не блокирует | Чинит в задаче |
| Техдолг сквозной (два модуля дублируют логику) | План выноса, граница | Исполнение по плану |
| Продуктовые требования | Комментарий «влияет на архитектуру» | Формализация в docs + код |
| Code review | **Обязателен** на: новый модуль, миграции с FK между доменами, изменения auth/PWA/offline | Обычный review остального |

**Правило:** если решение **нельзя откатить за один PR** или оно **меняет контракт для двух зон** — сначала архитектор (или совмещённая роль) фиксирует в техдоке.

## Сисадмин (04) vs архитектор (10)

| Тема | Архитектор | Сисадмин |
|------|------------|----------|
| Docker / compose / nginx | Требования: порты, домены, health | Конфиг, деплой, мониторинг |
| Очереди, Redis, cron | Семантика jobs, retry, idempotency | Запуск воркеров, алерты |
| Секреты и env | Какие ключи нужны приложению | Хранение, ротация |
| Масшта/HA | Не MVP; при росте — ADR «когда дробим» | Реализация |

## Директор (01)

- Архитектор **не** меняет scope MVP; эскалирует, если канон требует больше времени, чем план (например, полный offline для модуля).
- Крупные ставки (второй сервис, смена стека) — **предложение директору**, не самостоятельное решение.

## Совмещение ролей

На старте допустимо: **один человек = разработчик + архитектор**. Тогда перед merge значимых PR — **явная пауза**: «я как архитектор» пройти [чеклист](Чеклист%20перед%20новым%20модулем.md), не только «я как кодер».
