# Документация (вне Git приложения)

Каталог **`Документация/`** лежит **рядом** с репозиторием Laravel **`_telotron.ru/`** и держит материалы по продукту и исследованиям. Три верхних раздела:

| Раздел | Путь | Назначение |
|--------|------|------------|
| **Сырьё и исследования** | [`Сырьё и исследования/`](Сырьё и исследования/) | Черновики, интервью, бизнес-план, заметки — внутренняя структура произвольная. |
| **Решения** | [`Решения/README.md`](Решения/README.md) | Упорядоченные слои: от намерения до реализации и принципов работы. Словарь терминов: [`Решения/0-контекст-и-правила/Глоссарий.md`](Решения/0-контекст-и-правила/Глоссарий.md). |
| **docs (канон для кода)** | [`docs/`](docs/) | **Симлинк** на [`../_telotron.ru/docs/`](../_telotron.ru/docs/) — то же дерево на диске; версионируется только в Git репозитория **`_telotron.ru/`**, отдельной копии файлов нет. |

## Формальный поток разработки

- Начало работы с каноном: [`docs/README.md`](docs/README.md) — открывается тот же файл, что и [`../_telotron.ru/docs/README.md`](../_telotron.ru/docs/README.md), через симлинк.
- Правило Cursor для агента: **[`../.cursor/rules/dev-workflow.mdc`](../.cursor/rules/dev-workflow.mdc)** (при открытии только `_telotron.ru/` добавьте то же правило в `.cursor/rules/` репозитория приложения, если нужно).

## Правило переноса

Когда формулировка становится правилом для реализации — фиксируйте смысл в **`docs/`** (`_telotron.ru/docs/`), а в «Решениях» при необходимости оставьте ссылку на конкретный файл.
