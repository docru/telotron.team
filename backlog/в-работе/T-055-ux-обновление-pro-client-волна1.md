## Контекст

Тикет на внедрение результатов UX-аудита [T-057](../сделано/T-057-ux-аудит-экранов-с-дизайнером.md).

Цель: снизить трение в ключевых сценариях без большого редизайна:

1. сократить шаги до целевого действия;
2. выровнять консистентность паттернов между Pro/Client;
3. снизить перегруз на мобильных;
4. заложить guardrails под billing/partner/MAX.

---

## Scope (волна 1)

### A. Pro quick wins

1. **Клиенты → приглашение**
   - Сделать прямой action приглашения на `WorkspaceClientsPage`, без обязательного перехода через `Ещё`.
   - Допустимы варианты: inline panel / CTA с sheet.

2. **Расписание: progressive disclosure**
   - Разделить первичный и продвинутый шаги в create/edit flow.
   - На первом экране оставить только критичные поля: тип, начало, длительность.
   - Групповая специфика/уведомления — вторым шагом либо сворачиваемым блоком.

3. **Карточка клиента: уменьшить навигационную сложность**
   - Убрать избыточные одновременные переключатели.
   - Правило: не более 2 уровней выбора контекста одновременно.

### B. Client quick wins

4. **Дневник-хаб: снизить когнитивную нагрузку**
   - Пересобрать приоритеты section/subtab (по умолчанию — наиболее частый сценарий пользователя).
   - Ясная иерархия: что primary action сегодня.

5. **Главная: одна доминирующая цель «что делать сейчас»**
   - Уменьшить конкуренцию CTA и карточек на первом экране.
   - На верхней части экрана — один primary сценарий + 1 secondary.

### C. Сквозные системные

6. **Единые состояния**
   - Для ключевых страниц обеспечить единый шаблон `empty / error / loading / offline` + `next step CTA`.

7. **Терминология UI**
   - Провести сверку названий разделов, кнопок и lead-текстов (единый словарь в коде/доках).

8. **Guardrails расширяемости**
   - Зафиксировать и применить при добавлении новых экранов правила из T-057 (§5).
