## 01 · Директор

**Главное на Пилот:** доказать ценность на **реальных тренерах** и не сорвать **01.08** (stage billing к 31.07).

| Акцент | Почему |
|--------|--------|
| **Личный outreach** — первые 10–30 тренеров через доверие, не массовая реклама | Качество обратной связи важнее цифры «80 reg» до [T-004](../бэклог/T-004-оценка-привлечения-воронка.md) |
| **Activation > vanity reg** | Успех — тренер ведёт клиента, запись, день; иначе пилот бессмысленен |
| **Два плана спринта 1** — без них не размазывать dev и маркетинг | Урок спринта 0: календарь ≠ критический путь |
| **01.08 не двигаем** без формального go/no-go | Запасной план только продление триала, не «половина billing» |
| **MAX закрыт** — не возвращать в приоритет dev | Фокус коммерции и outreach |

**Не в фокусе:** найм, 100 платников (этап 2), полировка UI без evidence.

## 02 · Маркетолог (PO)

**Главное:** воронка и **упаковка к 01.08**, insight → backlog.

| Акцент | Почему |
|--------|--------|
| **Реалистичный темп** reg/нед — [T-004](../бэклог/T-004-оценка-привлечения-воронка.md), не KPI «с потолка» | Иначе план продвижения невыполним при solo |
| **Два направления** ([видение](../../02-Маркетолог/Инструкции/Продвижение%20—%20видение%20и%20направления.md)): личные связи **раньше** соцсетей | Ядро для партнёрки и кейсов |
| **Матрица тарифов** [T-005](../бэклог/T-005-матрица-функция-тариф.md) — черновик к **01.07** | Dev gating зависит от PO, не наоборот |
| **Insight brief раз в 2 нед.** | Минимум 3 за Пилот; без evidence — `идеи/` |
| **Коммуникация:** регистрация и триал **в продукте**; платежи — «с 01.08» | Не обещать то, чего нет в prod |

**В маркетинговом плане обязательно:** ритм касаний, KPI (reg, activation, интервью), связка A13 + `/i/{token}`, партнёры-ядро.

## 03 · Разработчик

**Спринт 1 (сейчас):** **вторично** — инвентарь идей, приоритет, **ТЗ на спринт 2** ([T-023](../в-работе/T-023-план-разработки-этап-1-пилот.md)). Код — **не в этом спринте**.

**Пилот в целом:** **billing + partner на stage к 31.07** при ~10–15 ч/нед.

| Акцент | Почему |
|--------|--------|
| **#1 в очереди** — [ADR-001](../../../_telotron.ru/docs/Техдок/00-мета/архитектурные-решения/ADR-001-scope-billing-partner-01-08.md) B1–B7, P1–P8 на **stage** | Блокер 01.08 |
| **Слайсы по спринтам** с оценкой часов — в T-023 | Иначе снова «не влезло» |
| **A2 (финансы клиента)** — после billing или явный минимум | [согласование A2/A12](../../../_telotron.ru/docs/Бизнес-требования/00-канон-mvp/согласование-бизнес-требований-mvp.md); не дефект сейчас |
| **P0 prod** вне очереди; мелочь без тикета — не делать | Правило retro спринта 0 |
| **Тесты с каждым слайсом** billing/partner | DoD этапа 2 — feature-тесты |

**В плане разработки обязательно:** таблица спринт × эпик × часы; зависимость от T-005; что **отложено** (онбординг-мастер, share, M4 notify).

**Не в фокусе сейчас:** MAX, public-витрина C7, in-app M4 UI.

## 04 · Сисадмин

**Главное:** prod **не мешает** пилоту.

| Акцент | Почему |
|--------|--------|
| **Стабильность** `fb8f6d9`+ : worker, scheduler, бэкапы ([T-003](../сделано/T-003-a1-a11-prod-runbook.md)) | Outreach на shaky prod убивает доверие |
| **Deploy по runbook**; откат известен | Dev не тратит ночи на инфру |
| **Секреты** MAX/TG/SMTP/будущая ЮKassa — только `.env`, не Git | Уже практика периода 0 |
| **Stage** для billing к июлю — заранее в плане T-023 | Не поднимать billing «в последнюю неделю» |

**В плане разработки:** отметить, нужен ли отдельный stage-контур для ЮKassa sandbox.

## 06 · Поддержка

**Главное:** обращения → **структура для PO**, не хаос в чате.

| Акцент | Почему |
|--------|--------|
| **M14 + созвоны** — топ болей **еженедельно** в PO | Питает insight brief |
| **Критерий activation** для пилота: «тренер сделал X» — согласовать с директором | Единое определение в T-004 / маркетинговом плане |
| **Классификация:** баг / известное / не баг / ТЗ | [Оформление дефектов](../../09-Тестировщик/Инструкции/Оформление%20дефектов.md) |
| **SLA разбора ≤ 48 ч** на P0/P1 пилота | Мало пользователей — каждый важен |

**В маркетинговом плане:** как собираем цитаты и кейсы для соцсетей (направление 2).

---

## 07 · Юрист

**Главное:** v1.0 **достаточно для пилота**; готовим **коммерцию**.

| Акцент | Почему |
|--------|--------|
| **v1.0 опубликован** — пилот без платежей ([T-007](../сделано/T-007-legal-privacy-v1-compliance.md)) | Не блокируем outreach |
| **v2.0 + внешний юрист → 01.08** | Оферта подписки, партнёрское, платежи |
| **Тексты подписки к 01.07** (цели этапа B3) | Параллельно dev billing |
| **В коммуникации** не обещать функции оплаты до go/no-go 01.08 | Риск претензий |
| **A2 финансы клиента** — отдельное согласование перед dev | Мед. и ПДн не смешать с «оплатой занятия» в мессенджерах |
| **T-021** РКН — не блокер | Статус в реестре по мере появления |

**В планах:** маркетинг — дисклеймеры триала; dev — gate юр. текстов перед prod billing.

## 09 · Тестировщик

**Главное:** **P0=0** на prod; приёмка billing — заранее.

| Акцент | Почему |
|--------|--------|
| **Регрессия критичных** перед каждым prod ([Критичные сценарии](../../09-Тестировщик/Инструкции/Критичные%20сценарии%20MVP.md)) | Gate A8 не разовый |
| **Критерий «можно звать тренера»** — зафиксировать в маркетинговом плане | Минимум: регистрация Pro, инвайт, запись клиента |
| **Billing/partner на stage** — отдельный набор feature-тестов (B7, P8) | Заложить в T-023 по спринтам |
| **Дефекты** только через `Дефекты/`; A2 «нет финансов» — **не баг** | [согласование MVP](../../../_telotron.ru/docs/Бизнес-требования/00-канон-mvp/согласование-бизнес-требований-mvp.md) |
| **Ручной smoke** после `build.*` bump | PWA cache |

**В плане разработки:** слот QA на каждый billing-slice (не «в конце июля»).

---

## 10 · Архитектор

**Главное:** billing+partner **без переписывания** Identity и Scheduling.

| Акцент | Почему |
|--------|--------|
| **Review до кода** — billing/partner по [модуль-оплаты-архитектура](../../../_telotron.ru/docs/Техдок/03-модули/модуль-оплаты-архитектура.md) | Самый рискованный контур года |
| **Границы модулей** `app/Modules/*`; проводки vs partner month | ADR-001 уже принят — не раздувать |
| **Stage webhook ЮKassa** — идемпотентность, как MAX/TG | Паттерн есть |
| **Gating** — один TariffResolver, не размазать по Vue | Зависит от T-005 |
| **Не микросервисы / не DDD-цирк** | Solo FTE |

**В плане разработки:** архитектурные вехи (миграции, сервисы, API) **до** UI gating.
