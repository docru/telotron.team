## Гипотеза

Если **не блокировать** установку PWA, бота MAX/TG, Passkey и длинный legal-wizard на критическом пути, а **отложить** их с понятными напоминаниями — **конверсия reg → первый вход в кабинет** вырастет без потери compliance (при корректной юр. схеме).

**Как проверим:** когорта пилота — % завершивших reg за сессию, % `connected` за 48 ч, отвал по шагам воронки (до/после).

## Идеи (канон формулировок)

### C-01 · PWA — не обязательна сразу

| | |
|--|--|
| **Суть** | Установка приложения (PWA) **не обязательна** на регистрации и в первом мастере. Периодически — мягкое напоминание «для удобства». |
| **Сейчас** | Шаг `install_pwa` в [T-060](../в-работе/T-060-pro-onboarding-k-p.md) уже **опциональный** в контуре К; install-баннер на `/more`. |
| **Целевое** | Не показывать install как блокер reg; **ритм напоминаний** (coach-bar / баннер / in-app после N дней). |
| **Риски** | Ниже retention у тех, кто не поставил PWA — компенсировать push/in-app. |

### C-02 · Бот — не обязателен при reg; вход через SMS

| | |
|--|--|
| **Суть** | Подключение бота MAX/TG **не обязательно** на регистрации. Постоянно (ненавязчиво) напоминать: «подключите канал — дойдут оповещения». **Регистрация** — через **SMS** (OTP на номер). |
| **Сейчас** | Канон: legal → **MAX/TG/e-mail** → Passkey ([auth](../../../_telotron.ru/docs/Бизнес-требования/02-модули/auth/регистрация-и-вход-max-passkey.md)). [T-064](../бэклог/T-064-auth-верификация-телефона-149-фз.md) — SMS как вариант A, PO рекомендовал MAX-first. |
| **Целевое** | SMS = primary proof на reg; bind бота — **после** входа, с persistent nudge в настройках / баннере. |
| **Открыто** | 149-ФЗ / юрист ([07-Юрист](../../07-Юрист/Вопросы%20юристу.md) — «SMS не используется» в каноне v1.0). Стоимость SMS, антифрод (F-05). |

### C-03 · Все соглашения на одной странице

| | |
|--|--|
| **Суть** | Один экран акцепта **всех** нужных документов (с прокруткой / раскрытием), вместо нескольких шагов wizard. |
| **Сейчас** | Client: 3 документа на шаге 1 + мед. согласие отдельно ([T-062](../бэклог/T-062-client-регистрация-единый-шаг-согласий.md) — шаг 4 **после** Passkey). |
| **Целевое** | **Один UX-экран**, но **отдельные** записи в `legal_acceptances` и audit (152-ФЗ) — согласовать с [07-Юрист](../../07-Юрист/). |
| **Зависимости** | [T-051](../бэклог/T-051-спринт2-qm-юр-тексты.md), [схема акцептов](../../07-Юрист/Схема%20юрдокументов%20и%20этапы%20акцепта%20(Pro%20·%20Client).md). |

### C-04 · Мастер необязателен; баннер сразу после reg

| | |
|--|--|
| **Суть** | Полноэкранный мастер **не обязателен**. Сразу после регистрации — **приглашающий баннер** («начать знакомство» / «настроить за 15 мин»), без блокировки кабинета. |
| **Сейчас** | [T-060](../в-работе/T-060-pro-onboarding-k-p.md) — coach-bar, «Пропустить»; [T-065](../бэклог/T-065-pro-onboarding-порядок-подготовка-клиент.md) — порядок шагов. |
| **Целевое** | По умолчанию — **расписание/главная**; баннер CTA → мастер; dismiss не скрывает навсегда (вернуть из «Ещё»). |
| **Связь** | F-13 в пожеланиях пилота. |

### C-05 · Passkey — понятно и можно отложить

| | |
|--|--|
| **Суть** | Ясное объяснение **зачем** Passkey (без жаргона). Возможность **пропустить** на reg с входом по SMS/OTP и **подключить Passkey позже** в настройках. |
| **Сейчас** | Passkey обязателен в каноне reg; copy минимальный (`passkeyLoginCta` в theme). |
| **Целевое** | Explainer + «Позже»; fallback auth до привязки Passkey; напоминание в профиле. |
| **Риски** | Безопасность, recovery — [техдолг auth](../../../_telotron.ru/docs/Техдок/00-мета/техдолг-auth-после-этапа-1.md). |
| **Связь** | F-14. |
