## Контекст

Сейчас ([T-010](../сделано/T-010-акцепт-медицинские-pd-client.md)):

- шаг 1 регистрации — **3** документа (`privacy_policy`, `terms_of_service`, `personal_data_consent`);
- `sensitive_data_consent` — **отдельный gate** при первом входе в «Дневник» / «Прогресс».

**Проблема UX:** на **главной** Client уже видны вода/шаги, но без мед. согласия API трекера блокируется → баннер «примите в Дневнике» раздражает и вводит в заблуждение.

**Решение (директор):** мед. согласие (`sensitive_data_consent`) — **шаг 4** регистрации Client: **сразу после Passkey**, **до** блокирующей анкеты. Не откладывать на первый вход в «Дневник» / «Прогресс».

> **Юридическое ограничение:** общий пакет (3 документа на шаге 1) и мед. согласие — **разные** акцепты в `legal_acceptances`; на шаге 4 — отдельный чекбокс + прокрутка текста (152-ФЗ ст. 9–10).

## Поток регистрации Client (канон)

```text
1. Телефон, имя, 3 юрдока + 18+
2. OTP (MAX / Telegram / e-mail)
3. Passkey
4. Согласие на мед. / спецкатегории ПДн  ← этот тикет
5. Анкета (профиль)
6. Кабинет (главная, трекер сразу работает)
```

**Pro** — без шага 4 (мед. gate не нужен).

## UX шага 4

| Элемент | Требование |
|---------|------------|
| Документ | `sensitive_data_consent` — полный текст, прокрутка обязательна |
| Чекбокс | «Ознакомлен(а) с текстом согласия» — не pre-checked ([ревью §7.3](../../07-Юрист/Ревью%20—%20согласие%20медицинские%20ПДн%20v1.0%20(2026-05-24).md)) |
| Кнопка | «Даю своё согласие» → `POST /me/legal-acceptances` → редирект на анкету |
| Отказ | Без акцепта в кабинет **не** пускать; «Назад» — только logout или явный отказ (без обхода API) |
| Копирайт визарда | Client: **«четыре шага»** (не три) |

После шага 4 — gate на `/appointments` и `/measurements` **не нужен** для новых пользователей; оставить **fallback** для legacy без акцепта.

## Backend
