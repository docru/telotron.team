## Контекст

Директор (25.06): пять направлений, чтобы **снизить сопротивление на входе** и поднять **конверсию reg**. Детали и формулировки — в [идее](../идеи/конверсия-регистрации-снижение-трения.md) (C-01…C-05).

Часть уже заложена в [T-060](../в-работе/T-060-pro-onboarding-k-p.md), [T-065](../бэклог/T-065-pro-onboarding-порядок-подготовка-клиент.md), [пожеланиях пилота](../../02-Маркетолог/Исследования/Пилот%20—%20пожелания%20тренеров/2026-06-24%20—%20сессия%20с%20тренером.md).

**Принцип эпика:** на критическом пути reg — **минимум обязательных шагов**; всё «для удобства» (PWA, бот, Passkey, мастер) — **опционально сейчас + напоминания потом**, без потери compliance.

## Критерии готовности эпика (высокий уровень)

- [ ] Зафиксирован **целевой поток reg** Pro (и отдельно Client) в [каноне auth](../../../_telotron.ru/docs/Бизнес-требования/02-модули/auth/регистрация-и-вход-max-passkey.md).
- [ ] **C-01, C-04** — в проде: reg без обязательного PWA и без блокирующего мастера; баннер + напоминания.
- [ ] **C-05** — explainer Passkey + skip с fallback (если директор утвердил).
- [ ] **C-02** — решение go/no-go SMS + post-reg bot nudge; при go — реализация + антифрод.
- [ ] **C-03** — юрист подписал схему «один экран / несколько акцептов»; UI в reg.
- [ ] Метрика: измеряем отвал по шагам **до/после** (воронка [T-050](T-050-спринт2-воронка-метрики.md)).

---

## Рекомендуемый порядок (до реализации)

1. **Решение директора** (30 мин): SMS-first (C-02) да/нет; Passkey optional (C-05) да/нет.
2. **Юрист** (C-03, C-02): один экран согласий; SMS как proof номера.
3. **Быстрые UX** без смены auth: **C-04 + C-01** (баннер, PWA nudge) — можно параллельно T-065.
4. **Auth-срез** C-02 + C-05 — отдельный тикет после legal.
