## Контекст

Сейчас при регистрации Pro/Client:

- номер **вводится** на шаге 1 и нормализуется (E164) — **без SMS/звонка**;
- подтверждение — **MAX / Telegram** (bind бота) или **e-mail** (OTP);
- вход — **Passkey**; телефон используется как **логин** для поиска учётки.

**Факт:** владение номером **не доказывается**. Пользователь может указать любой `+7…` и пройти регистрацию через Telegram.

**Риски:**

| Область | Проблема |
|---------|----------|
| **149-ФЗ / 199-ФЗ** | Нельзя честно позиционировать «аутентификация по российскому номеру телефона» — номер не верифицирован |
| **Telegram** | При TG-only путь фактически ближе к «иностранный канал + Passkey», чем к телефону РФ |
| **Продукт** | Захват чужого номера в профиле, ненадёжное восстановление |
| **Документы** | В политике/РКН телефон как контакт при отсутствии `phone_verified` |

Канон текущей модели: [регистрация-и-вход-max-passkey](../../../_telotron.ru/docs/Бизнес-требования/02-модули/auth/регистрация-и-вход-max-passkey.md) · техдолг: [техдолг-auth-после-этапа-1](../../../_telotron.ru/docs/Техдок/00-мета/техдолг-auth-после-этапа-1.md).

## Решение директора (выбрать до разработки)

Один из вариантов — **обязательный gate** перед закрытием тикета:

| ID | Вариант | Суть | SMS |
|----|---------|------|-----|
| **A** | OTP **на номер** (SMS или flash-call) | Классическая верификация; сильный compliance | Да / провайдер |
| **B** | **MAX обязателен** на регистрации; телефон вторичен до верификации | Российский контур; bind MAX как доказательство канала | Нет |
| **C** | **`phone_verified_at`** + честные тексты; TG/e-mail — канал, не «телефон РФ» | Минимум кода; не закрывает 149-ФЗ полностью | Нет |
| **D** | **VK ID / Яндекс ID / ЕСИА** (опционально или обязательно) | Запас по 149-ФЗ; отдельная интеграция | Нет |

**Рекомендация PO (до решения директора):** для пилота без SMS — **B + C** (MAX приоритет, флаг верификации, правка доков); **D** — этап 2; **A** — если юрист потребует явный OTP на номер.

## Технические требования (после выбора варианта)

### Общее (все варианты)

- [ ] Поле **`phone_verified_at`** (nullable timestamp) на `users` или `profiles`; в `GET /me` — `phone_verified: bool`.
- [ ] Не считать регистрацию «по телефону РФ» завершённой, пока `phone_verified_at` null (для audit/юриста).
- [ ] Миграция: существующие пользователи — `phone_verified_at = null` (legacy).
- [ ] Документ для юриста: фактическая схема auth (1 абзац) + выбранный вариант.

### Вариант A (SMS / flash-call)

- [ ] Провайдер (российский); OTP на номер **до** или **после** bind MAX/TG.
- [ ] Rate limit, throttle, audit.
- [ ] Feature-тест: без кода на номер — регистрация не завершается.

### Вариант B (MAX обязателен)

- [ ] Регистрация Pro/Client: канал MAX **по умолчанию**; TG/e-mail — только fallback с явной маркировкой.
- [ ] Уточнить у MAX: bind доказывает владение номером или только чатом.
- [ ] Если MAX не доказывает номер — комбинировать с A или D.

### Вариант C (честные тексты)

- [ ] Политика / оферта / A13: «номер указывается пользователем; подтверждение — Passkey и канал MAX/Telegram/e-mail» до появления верификации номера.
- [ ] UI: не писать «подтверждённый телефон», пока `phone_verified_at` null.

### Вариант D (российская ИС)

- [ ] Spike: VK ID или Яндекс ID как доп. способ; телефон из ответа провайдера → `phone_verified_at`.
- [ ] Отдельный подтикет при выборе D.

## Compliance-чеклист (149-ФЗ / 199-ФЗ)

- [ ] Нет кнопок Google / Apple / GitHub OAuth (аудит лендингов + prod).
- [ ] В матрице способов входа зафиксирован **фактический** канон (не «верифицируем телефон», если не верифицируем).
- [ ] Заключение юриста по схеме **после** выбора варианта A–D.
- [ ] При штрафном контексте (199-ФЗ с 07.2026) — приоритет российских каналов (MAX > TG > e-mail).

## Критерии готовности

- [ ] Директор + юрист: выбран вариант A/B/C/D (запись в журнале тикета).
- [ ] Реализован выбранный контур; `phone_verified_at` в API.
- [ ] Новая регистрация не позволяет считать номер подтверждённым без выбранного механизма.
- [ ] Feature-тесты на happy path и отказ без верификации.
- [ ] Обновлены техдок auth + при необходимости legal-sources.
- [ ] Ручной прогон: регистрация Client/Pro с «левым» номером + TG — поведение соответствует решению.

## Вне scope

- Массовая SMS-рассылка маркетинга.
- Замена Passkey.
- Admin/Filament (отдельный контур: телефон + пароль).
