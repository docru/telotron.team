## Контекст

Запрос директора: текущий UX ощущается неудобным и непоследовательным для **Pro** и **Client**, требуется системный аудит и практичный план улучшений без полного редизайна.

Фокус аудита (по интервью с директором):

1. **Слишком много шагов** до целевого действия.
2. **Непоследовательный UI** между экранами.
3. **Перегруженность на мобилке** (плотность, конкуренция блоков, сложные формы).

Источник идеи: [`ux-аудит-с-дизайнером-модульная-сетка.md`](../идеи/ux-аудит-с-дизайнером-модульная-сетка.md).

## 2) Единый шаблон оценки

Для каждого сценария фиксировать:

- цель пользователя за 1–2 минуты;
- фактический путь (шаги/клики/переходы);
- барьеры (навигация, плотность, терминология, состояния);
- влияние на бизнес (частота, риск потери/ошибки);
- оценку сложности (S/M/L) и приоритет (P1/P2/P3);
- тип решения: quick win / системное правило / архитектурный долг.

## 4) Топ улучшений (quick wins + системные)

### Quick wins (P1, 1 спринт)

1. **Прямой entrypoint приглашения** из `Клиенты` (убрать крюк через `Ещё`).
2. **Упростить шапку дневника Client:** одна главная цель в разделе по умолчанию.
3. **Снизить плотность Pro-календаря:** progressive disclosure в модалках (базовые поля сначала).
4. **Единый паттерн empty/error/offline с CTA next step** на ключевых экранах.
5. **Терминологический словарь UI** и сверка label/sectionLead с роутами.

### Системные (P1/P2, 2–3 спринта)

6. Каноническая **карта сценариев Pro/Client** (1 целевое действие на экран/контекст).
7. Общие **правила навигационной глубины** (не более 2 переключателей уровня одновременно).
8. Единый **UX-пакет расширяемости** для billing/partner/MAX.
9. Стандартизация **mobile density**: лимиты блоков «над фолдом», приоритет CTA.
10. Минимальный **UX smoke-чеклист** перед релизом для каждой зоны.

---

## 5) UX-правила расширяемости (MVP + near roadmap)

Обязательные guardrails:

1. **Один тип действия — один паттерн UI** во всех зонах.
2. **Не добавлять новый паттерн**, если задачу покрывает текущий компонент/flow.
3. **Mobile-first при конфликте плотности:** читаемость и next step важнее насыщенности.
4. Для каждого нового экрана обязательны: **empty / error / loading / offline + явный CTA**.
5. Новые модули (billing/partner/MAX) встраиваются через существующий shell, а не отдельный «параллельный мир».
6. Верхнеуровневый экран = **одна первичная цель**, вторичные — в «Ещё», sheet или вторичный шаг.
7. Избегать одновременных «вкладки + подвкладки + фильтр-чипы» на одном первом экране без явной причины.
