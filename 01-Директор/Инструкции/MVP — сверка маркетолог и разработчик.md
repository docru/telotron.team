# MVP — сверка маркетолог и разработчик

## Статус
- Дата: 09.05.2026
- Сверка: актуальный контур `02-Маркетолог` vs модульный контур `03-Разработчик`
- Итог: базовая синхронизация достигнута, ключевые противоречия по продажникам/менеджерам закрыты.

## Зафиксировано (решения)
1. В MVP **нет отдельной роли продажника**.
2. На первом этапе продажи ведутся **силами основателя**.
3. Рост после старта идёт через **партнёрку**.
4. Отдельная роль **поддержки** (бывш. аккаунт-менеджеры) появляется после **500-1000 платящих**.
5. В фазе 1 MVP используем только клиентские модули:
   - №1 клиент
   - №2 специалист
   - №3 (маркетолог) — не входит в MVP фазы 1
   - №4 (руководитель) — не входит в MVP фазы 1
6. Для MVP утверждён модульный срез: **M0, M1, M2, M4, M5, M13, M14**.
7. Для MVP утверждена маркетинговая рамка:
   - тарифы **Light 0 / Pro 5000 / Max 10000**;
   - промо **2 месяца Max**, затем переход на **Light**;
   - на **Light** ограничиваем **функциональность**, а не число клиентов;
   - на старте только подписка, без **split**;
   - продажи на первом этапе силами основателя, рост дальше через партнёрку.
8. Цены **Pro 5000 / Max 10000** утверждены как рабочая гипотеза MVP:
   - сохраняем без изменений на старте;
   - фокус в MVP на упаковке модулей и ценности, а не на смене цен;
   - пересмотр цен делаем после накопления первых платящих и фактических метрик конверсии/удержания.
9. Модель MVP: **один специалист = одна практика** (модель организации с несколькими специалистами отложена).
10. Каналы напоминаний в MVP:
   - по умолчанию **in-app** + **Telegram**;
   - если пользователь авторизован через email — дублируем напоминание на email.
11. Канал **MAX** зафиксирован как стратегический приоритет:
   - MAX рассматривается как более устойчивый канал для долгого горизонта (в т.ч. с учётом рисков блокировок и доступности);
   - реализовать MAX нужно как можно раньше после запуска MVP;
   - в сам MVP MAX не включаем как обязательный блокер, так как реализация сложнее (сайт/интеграция и организационные шаги).
12. Группы в MVP: включаем **облегчённый сценарий**, без полного отдельного модуля.
   - **Входит в MVP:** в календаре есть тип/режим группового занятия; к одному занятию можно привязать несколько клиентов; базовая отметка посещения/статуса.
   - **Не входит в MVP:** групповой чат, отдельные групповые оплаты/тарифы, расширенная аналитика по группам.
   - **Критерий готовности:** специалист может создать одно групповое занятие, назначить в него нескольких клиентов, а клиенты видят это занятие в своих расписаниях.
13. M0 (Платформа и роли) — уточнение для MVP:
   - основной логин: **телефон**;
   - email: **опционально**, если пользователь выбирает подтверждение/восстановление через email;
   - после регистрации клиента обязательна короткая анкета: **ФИО, пол, дата рождения**.
14. M1 (Карточка клиента / CRM) — уточнение для MVP:
   - специалист **не создаёт клиента вручную**;
   - специалист отправляет клиенту **ссылку-приглашение** (нужен лёгкий механизм "поделиться ссылкой");
   - клиент проходит по ссылке, регистрируется и автоматически появляется в кабинете тренера;
   - ссылки приглашения клиентов и партнёрские ссылки работают на **едином механизме**;
   - формат ссылок: **короткие постоянные ссылки**, пригодные для размещения на внешних ресурсах и пересылки.
   - статусы клиента в MVP **убираем** (переносим на следующий этап).
15. M2 (Календарь и журнал) — утверждено для MVP:
   - специалист создаёт занятие (дата/время/клиент);
   - клиент видит предстоящие занятия;
   - специалист может перенести/отменить занятие;
   - в базовом сценарии нет критических конфликтов по времени;
   - групповые сценарии обсуждаются отдельно.
16. M4 (Напоминания) — утверждено для MVP:
   - для занятия формируется напоминание;
   - напоминание отправляется в утверждённые каналы MVP;
   - фиксируется статус доставки (успех/ошибка);
   - при временной ошибке канала есть повторная отправка;
   - тренер перед занятием видит в календаре статус уведомления: **дошло / прочитано**.
17. M5 (Планы тренировок и питания) — утверждено для MVP:
   - встроенной библиотеки шаблонов в MVP нет;
   - конструктор планов в MVP не делаем (переносим на следующий этап);
   - специалист создаёт план как прикреплённый файл (**PDF/DOC/XLSX**);
   - специалист назначает файл плана клиенту;
   - клиент может открыть план в режиме чтения и скачать файл;
   - храним минимум метаданных: кто назначил и дата текущего назначения.
18. Требования 152-ФЗ (обязательный контур):
   - **до подключения реальных пользователей (период I):** подать **уведомление в Роскомнадзор** об операторе ПДн — [канон](../../../_telotron.ru/docs/Бизнес-требования/04-комплаенс-и-аудит/уведомление-роскомнадзор-оператор-пд.md);
   - поручить юристу подготовить тексты для акцепта в форме регистрации;
   - добавить механизм обезличивания данных клиента;
   - добавить логирование действий, связанных с персональными данными.
19. Партнёрка в MVP — минимальный контур:
   - в MVP обязательно фиксируем атрибуцию: кто подключился по чьей ссылке;
   - реферальная ссылка работает с первого дня (для раннего органического привлечения);
   - начисления, балансы и выплаты партнёрам в MVP не включаем;
   - списки подключённых и расширенные партнёрские экраны — после MVP.
20. Трекер в MVP — отдельный упрощённый модуль (**M7-lite**):
   - тренировка: чекбокс;
   - приём пищи: текстовое поле;
   - шаги: числовое поле;
   - вода: числовое поле;
   - сон в MVP не делаем;
   - модуль без сложной аналитики и рекомендаций на этапе MVP.
21. M13 (Динамика прогресса) — утверждено для MVP:
   - ключевая метрика по умолчанию: **вес**;
   - вносить данные может **только клиент**;
   - история значений отображается списком;
   - есть простой график динамики;
   - без сложной аналитики, прогнозов и рекомендаций;
   - фото-фиксация замеров в MVP не входит (переносится существенно позже, в т.ч. из-за юридических требований к чувствительным данным).
22. M14 (Обратная связь) — утверждено для MVP:
   - форма обратной связи доступна в приложении клиента и кабинете специалиста;
   - тип обращения: **ошибка / неудобно / идея**;
   - текст обращения обязателен;
   - автоматически передаётся контекст: роль, версия приложения, время, id пользователя;
   - после отправки пользователь видит подтверждение "принято";
   - команда получает обращения в единый канал обработки;
   - SLA-кабинет, тикет-система и внутренняя переписка по обращению в MVP не входят.

## Текущее соответствие
- По стратегии роста и монетизации: высокое.
- По границам MVP: выровнено после правок.
- По ролям и клиентским приложениям: выровнено (продажник исключён, поддержка отложена до порога).

## Что исправлено как техдолг
- Убраны формулировки про обязательный MVP-модуль для продажника/менеджера.
- Обновлены термины в модульных документах (`маркетолог`, `поддержка`).
- Обновлены устаревшие ссылки после миграции из `Документация` в `Команда`.

## Остаточные вопросы (на отдельную сессию)
1. Когда именно запускать модуль №3 (маркетолог): до или после стабилизации фазы 1.
2. Какой минимальный набор экранов №3 нужен для партнёрки (если запускать).
3. При каком фактическом пороге (внутри коридора 500-1000) вводить роль поддержки.
4. План внедрения MAX как приоритета post-MVP: этапы, зависимости, ответственность.
5. Юридический пакет 152-ФЗ для MVP: **уведомление РКН**, тексты акцепта, обезличивание, аудит действий с ПД.
6. Технический дизайн партнёрки post-MVP: начисления, отчётность, выплаты.
