# Глоссарий проекта Телотрон

Единый словарь терминов для документации, продуктовых обсуждений и кода. При появлении нового понятия сначала фиксируем здесь, затем используем в остальных файлах без перефразирования.

**Версия:** v0.15 (живой документ).

---

## Платформа и приложения (обязательное различие)

| Термин | Определение |
|--------|-------------|
| **Платформа Телотрон** (или **платформа**) | Весь программный комплекс продукта: серверная часть, API, веб-доступ по зонам (доменам), интеграции, общая модель пользователей и данных — всё, что не сводится к одному установочному пакету на телефоне. |
| **Приложение** | Только клиент, который пользователь **устанавливает на телефон** из магазина приложений (или эквивалентная нативная/кроссплатформенная установка на мобильное устройство). Слова «приложение» в документации и требованиях в этом смысле не используем для веб-страниц и админки без отдельной оговорки. |
| **Телотрон** | Название продукта/сервиса; в разговорной речи часто совпадает с «платформой». Если важна точность (договор, техдок, метрики) — пишем **платформа** для комплекса и **приложение** только при установке на телефон. |
| **Приложение тренера** | Мобильное приложение для роли «тренер» (основной пользователь-оплатчик на старте), устанавливаемое на телефон. |
| **Приложение клиента** | Мобильное приложение для конечного клиента тренера, устанавливаемое на телефон. |
| **Приложение менеджера** | В коде и техдоке может означать клиент для роли **сопровождения** тренеров. В бизнес-модели отдельных **менеджеров по продажам** нет; **аккаунт-менеджеры** планируются **после ~500–1000 платящих**. На ранних этапах приложение под эту роль **не** обязательно; процессы — админка, таблицы. |
| **Веб-клиент** / **зона** | Доступ к возможностям платформы через браузер (в т.ч. зоны Pro, Client, публичная зона, админка). Не называть «приложением» без явного уточнения, если это не установка из стора на телефон. |
| **Клиентский модуль** | Отдельная сборка интерфейса под роль (часто — мобильное приложение из стора; может быть веб-клиент). В этапности MVP нумеруются №1–№4; не смешивать с ролью «клиент» (человек). |

### Исключение: слово «приложение» про репозиторий

В текстах про структуру репозитория допускаются формулировки **«репозиторий приложения»**, **«Git приложения»** — это про кодовую базу Laravel в `_telotron.ru/`, а не про мобильный клиент из стора.

## Роли людей (бизнес)

| Термин | Определение |
|--------|-------------|
| **Тренер** | Специалист, ведущий клиентов; плательщик **месячной подписки** на платформу. Целевой сегмент на старте: **индивидуалы**; **студии** — не раньше порядка **200–500 платящих** и найма доп. программиста (см. бизнес-модель §6, §8). |
| **Клиент** | Человек, который занимается у тренера и пользуется платформой (приложение клиента и/или веб-клиент зоны Client) по приглашению/связке с тренером. |
| **Менеджер по продажам** | В текущей оргмодели **не используется**: продажи ведут **основатель** и **партнёрка** (тренеры приводят тренеров). |
| **Аккаунт-менеджер** (в бизнес-модели Телотрона) | Роль людей на **коннект и удержание**, планируемая **после** порядка **500–1000 оплаченных** подключений (не холодные продажи). Вознаграждение и KPI — по регламенту при найме (см. бизнес-модель §3). Не путать с «менеджером» как руководителем проекта. В техдоке слово **менеджер** в именах зон/ролей может сохраняться исторически. |
| **Головной партнёр (компании)** | Условная вершина **дерева партнёров**: тренер, пришедший **напрямую** (открытый вход без кода коллеги), учитывается в партнёрке как привлечённый «под компанию»; выручка с подписки — у компании. Формулировки в оферте и UI — без введения пользователя в заблуждение (см. бизнес-модель §12.1). |
| **Руководитель** (в бизнес-модели) | Роль владельца/управления компанией: юнит-экономика, продажи, разработка, качество, безопасность — обычно внутренние дашборды, не путать с ролью в Filament без контекста. |

## Тарифы абонплаты тренера (канон имён на русском)

Имена **тарифов подписки** на платформу — только в форме ниже (в UI, оферте, `docs/`, бизнес-черновиках при переносе). Технические ключи в коде могут быть латиницей (`light`, `profi`, `max`) — маппинг в техдоке при реализации.

| Термин | Определение |
|--------|-------------|
| **Тариф «Лайт»** | Нулевая абонплата (**0 ₽**/мес в текущей модели); ограничения по функциям — по продуктовой матрице. Канон имени: **Лайт**. |
| **Тариф «Профи»** | Основной платный уровень (**5000 ₽**/мес на дату фиксации в [`модуль-оплаты-платформа.md`](../../../_telotron.ru/docs/Бизнес-требования/модуль-оплаты-платформа.md)). Канон имени: **Профи**. **Не путать** с **зоной Pro** (поддомен и веб-клиент для роли тренера) — это разные сущности. |
| **Тариф «Максимальный»** | Старший платный уровень (**10 000 ₽**/мес). Канон имени: **Максимальный**. |

## Термины воронки и экономики

| Термин | Определение |
|--------|-------------|
| **Платящий тренер** | Тренер с активной оплатой доступа к сервису в данный период (не только зарегистрированный). |
| **Подключённый тренер** | Тренер с учётной записью в системе; может быть на триале или без оплаты — уточнять в метриках отдельно от «платящий». |
| **Триал** | Пробный период доступа к платным возможностям; параметры (срок, карта) задаются отдельным решением. |
| **Удержание (тренера)** | Продолжение оплаты и/или регулярного использования продукта тренером; при появлении **аккаунт-менеджеров** — KPI в договорённостях с ними. |
| **Удержание (клиента тренера)** | Снижение оттока клиентов у тренера за счёт инструментов Телотрона (чат, контроль, напоминания и т.д.) — продуктовая цель, не путать с удержанием тренера. |
| **База (партнёрка)** | Сумма платежа привлечённого тренера за период начисления, по которой считаются доли (какие списания входят в базу, возвраты — в [`модуль-партнерка.md`](../../../_telotron.ru/docs/Бизнес-требования/модуль-партнерка.md)). |
| **Партнёрский внутренний счёт** | Учётный баланс тренера-партнёра в платформе: **начисление вознаграждения по партнёрке — 1-го числа** за **истёкший месяц**; **вывод и иные списания** с этого счёта — **только после** подписания **партнёрского договора**. Сумма **свыше порога S** **не сгорает**, копится на счёте. До договора часть начислений может **уменьшать** плату за абонплату (скидка) — см. [`модуль-партнерка.md`](../../../_telotron.ru/docs/Бизнес-требования/модуль-партнерка.md). |
| **Партнёрка** | Модуль платформы: **тренер приглашает тренера** (многоразовые ссылки, **кампании**), **без перепривязки**; **доли:** **L1 — 20%** от базы; **L2 — 25%** от начисления L1 (**5%** от базы); **L3 — 50%** от начисления L2 (**2,5%** от базы), L3 — после договора. Учёт через **партнёрский внутренний счёт** и порог **S** — см. [`модуль-партнерка.md`](../../../_telotron.ru/docs/Бизнес-требования/модуль-партнерка.md). Не смешивать с **менеджер → тренер** в Filament. |
| **Атрибуция** | В партнёрке — правило, по которому тренер попадает в **дерево** (реферер-тренер, **головной партнёр** компании при прямом входе и т.д.) для учёта начислений. Для **аккаунт-менеджеров** (когда появятся) — отдельное правило закрепления за человеком на удержание, не смешивать с партнёрской атрибуцией без явного решения. |
| **Split** | Доля платформы с платежа клиента → тренеру при оплате через платформу. **На текущем этапе не планируется**; термин оставлен для будущих обсуждений и техдока. |
| **Revshare** / **доля сопровождения** | Ориентир в обсуждении **20–30%** — к пересмотру при найме **аккаунт-менеджеров**; не путать с **партнёрскими** начислениями тренер → тренер (§6.3 бизнес-модели). |

## Регистрация и доступ

| Термин | Определение |
|--------|-------------|
| **Клиентская ссылка** | Пригласительная ссылка тренера для приглашения **клиентов** в продукте (технически: `invite_tokens.purpose = client_onboarding`). В MVP у тренера должна существовать минимум одна активная клиентская ссылка. |
| **Партнёрская ссылка** | Пригласительная ссылка тренера для приглашения **партнёров/тренеров** (технически: `invite_tokens.purpose = specialist_referral`). |
| **Пригласительный токен** | Секрет, который платформа вкладывает в **ссылку** (например `?ref=…`, путь `/ref/…`) или принимает как **вводимый код**; служит для однозначной **привязки** пользователя к другой стороне (клиент → тренер, тренер → менеджер, менеджер → учёт руководства). Обычно **одноразовый** или с ограниченным сроком действия. На веб-клиенте до регистрации токен из ссылки часто сохраняют в **`localStorage`** того же origin, чтобы не потерять при установке **PWA**. **Не путать** с **кодом подтверждения регистрации** (канал **MAX** или **e-mail** при создании учётной записи — см. [`регистрация-и-вход-max-passkey.md`](../../../_telotron.ru/docs/Бизнес-требования/регистрация-и-вход-max-passkey.md)). |
| **Регистрация по цепочке** | Базовая модель: **руководство** формирует в **админке Filament** пригласительную ссылку с токеном для **менеджера** → менеджер выдаёт ссылку/код **тренеру** → тренер выдаёт ссылку/код **клиенту**. Канон: [`приглашения-менеджер-тренер-и-руководство-filament.md`](../../../_telotron.ru/docs/Бизнес-требования/приглашения-менеджер-тренер-и-руководство-filament.md), [`клиент-привязка-к-тренеру-код-и-несколько-тренеров.md`](../../../_telotron.ru/docs/Бизнес-требования/клиент-привязка-к-тренеру-код-и-несколько-тренеров.md). |
| **Свободно пришедший тренер** | Тренер, зарегистрировавшийся через **открытый вход** без реферального кода коллеги; в партнёрке учитывается под **головным партнёром** компании (см. бизнес-модель §12.1). |
| **Назначение менеджера (рандом)** | Устаревшая формулировка; **рандомного закрепления за менеджером по продажам** нет. Для прямого входа используется логика **головного партнёра**; закрепление за **аккаунт-менеджером** — только после введения роли и регламента. |
| **Открытое окно входа** | Сценарий UI/процесса, позволяющий начать регистрацию без кода от менеджера/тренера (с последующим назначением или выбором). |
| **Витрина тренеров** | Публичный или клиентский сценарий **выбора тренера** свободно пришедшим клиентом. **Перспектива:** на **странице ввода кода** тренера после регистрации (если клиент пришёл не по пригласительной ссылке) — возможность выбрать тренера с витрины вместо или в дополнение к ручному коду; детали UX — при реализации. |

## Зоны развёртывания (технический канон)

Согласовано с [`docs/Бизнес-требования/многозонная-платформа.md`](../../../_telotron.ru/docs/Бизнес-требования/многозонная-платформа.md) в репозитории приложения.

| Термин | Определение |
|--------|-------------|
| **Публичная зона** | Основной домен: витрина, маркетинг, заглушки; без лишних ссылок на закрытые зоны платформы без необходимости. |
| **Админка** | Поддомен администрирования: Filament, RBAC (Shield и др.), доступ для персонала с ролями. |
| **Зона Pro** | Поддомен для специалиста (тренера): отдельная сессия; развитие сценариев — по техдоку. |
| **Зона Client** | Поддомен для клиента: отдельная сессия; развитие сценариев — по техдоку. |
| **SSO между зонами** | Сквозной единый вход между поддоменами **не** предполагается на текущем этапе (явное решение архитектуры). |

В техдоке и договорённостях по доменам используйте термины **зона** / **веб-клиент**; слово **приложение** — только в смысле установки на телефон (см. раздел выше).

## Программы тренировок и календарь

Канон продукта: [_telotron.ru/docs/Бизнес-требования/02-модули/m5-programmy-trenirovok/программы-тренировок.md](../../../_telotron.ru/docs/Бизнес-требования/02-модули/m5-programmy-trenirovok/программы-тренировок.md). Разморозка границ MVP: [разморозка-mvp-программы-и-трекеры](../../../_telotron.ru/docs/Бизнес-требования/00-канон-mvp/разморозка-mvp-программы-и-трекеры.md).

| Термин | Определение |
|--------|-------------|
| **Упражнение** | Элемент **базы тренера** — название движения («Присед», «Жим лёжа»). Не путать с **заданием**. |
| **Задание** | **Упражнение** + параметры для клиента: подходы, повторения, вес (кг), время; опционально **отдельные параметры на каждый подход**. |
| **Комплекс** | Упорядоченный список **заданий** (шаблон тренировки **без даты**). Слово **«занятие»** для комплекса **не используем**. |
| **Программа упражнений** | Упорядоченный список **комплексов**; в MVP без привязки к дням недели. |
| **Назначение программы** | Снимок программы у клиента: одна **активная**, прежние в **истории** (только план, без факта выполнения в архиве). |
| **Выполнение комплекса** | Прохождение клиентом **одного** комплекса; незавершённое можно **продолжить**; после полного завершения следующее «Начать» — **новое** выполнение (`workout_complex_executions`). |
| **Занятие (календарь M2)** | Встреча **тренер–клиент** с датой и временем. **Не** комплекс и **не** выполнение комплекса. |
| **Раздел «Расписание» (Pro)** | **Название в UI** зоны **Pro** для модуля M2: занятия тренера, слоты, перенос/отмена. Маршрут `/schedule`, нижняя вкладка и заголовок экрана — **«Расписание»**. В техдоке модуль может называться «календарь и запись»; в интерфейсе тренера слово **«Календарь»** для этого раздела **не используем**. |
| **Раздел «Дневник» (Client)** | **Название в UI** зоны **Client** для сводного экрана: тренировки, питание, трекеры, запись на занятия (feed + вкладки). Маршрут `/appointments`, нижняя вкладка и заголовок — **«Дневник»**. В техдоке — «календарь клиента» / feed; в интерфейсе клиента слово **«Календарь»** для **названия раздела** **не используем**. Внутри раздела режимы «лента / день / неделя» — **режимы просмотра**, не название раздела. |
| **Дневник питания** | Подраздел внутри **«Дневник»** (Client): приёмы пищи, текст + фото — не путать с названием всего раздела **«Дневник»**. |
| **Согласие на обработку медицинских данных** | Отдельный юридический документ и **отдельный экран акцепта** для **особой категории** ПДн (медицинские/здоровье: замеры, трекер, дневник и т.п.). **Не** при регистрации; **при первом входе** клиента в раздел **«Дневник»** или **«Прогресс»** (измерения). Кнопка **«Даю своё согласие»** + прокручиваемый текст; фиксация на сервере (время, IP, user-agent, идентификатор устройства). См. [T-009](../../backlog/сделано/T-009-согласие-медицинские-pd-черновик.md), [T-010](../../backlog/сделано/T-010-акцепт-медицинские-pd-client.md). |
| **M5 (legacy)** | Прежний срез MVP: план **файлом**. Для **тренировок** заменён конструктором программ; для **питания** в MVP остаётся **файл**. |
| **M7-lite (legacy)** | Галка «тренировка за день» — **снята**; учёт нагрузки — через **выполнения комплексов**. |
| **Краткий онбординг** | После регистрации: пригласить клиента, (группа), занятие; **можно пропустить** целиком. |
| **Полный онбординг** | Программа, назначение, партнёрка, обратная связь, файл питания — **необязателен**, доступен позже. |

## Этапы поставки (общие слова)

| Термин | Определение |
|--------|-------------|
| **MVP** | Минимально жизнеспособный набор функций для проверки гипотезы с реальными пользователями; границы фиксируются в продуктовых документах после исследований. |
| **Пилот** | Ограниченный запуск на малой выборке пользователей с измерением метрик. |
| **Волна продаж** | Плановый период массового подключения тренеров с целевыми числами (например, платящих к дате). |

---

## Обратная связь

| Термин | Определение |
|--------|-------------|
| **Форма обратной связи** | Сквозной элемент: во всех **мобильных приложениях** (тренер, клиент; при появлении роли — аккаунт-менеджер) и в основных **веб-клиентах** платформы для соответствующих ролей. Заметный сбор жалоб, идей, ошибок; не прячется в «подвале» как единственная точка входа. |
| **SLA обратной связи** | Согласованные сроки ответа и обработки; пока не зафиксированы — см. открытые вопросы. |

## Как пользоваться глоссарием
